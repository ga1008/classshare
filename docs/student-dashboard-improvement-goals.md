# 学生端首页改进目标文档

> 版本：v1.0（2026-06-12）
> 范围：学生角色登录后的 `/dashboard` 页面（`templates/dashboard.html` 学生分支 + `classroom_app/services/dashboard_service.py` 的 `_build_student_dashboard_context` / `_build_student_cockpit`）
> 总体方向：**易用性、教学创新性、美观性**。所有改动遵循既有约定：单一焦点、副栏弱化、`dashboard_agenda_events` 统一数据源、CSS 追加到 `ui-system.src.css` 底部 polish 层、新交互优先"服务端 partial + 小型自包含 JS 模块"。

---

## 现状基线

学生首页当前结构（自上而下）：

1. **Daily cockpit 驾驶舱**（`student-cockpit`）：左侧"待办与提醒" agenda 小组件，右侧今日统计四格 + "下一步"优先级清单（≤3 条）+ "课程脉搏"（修为进度条）。
2. **课堂列表**：搜索 + 筛选标签 + `课程列表 / 我的日程` 展示切换 + 课堂卡片网格。
3. **学期日历**：可拖动周历 + 聚焦周待办。
4. **右侧栏**：快捷入口（React 岛）、优先处理（focus）、最近动态、登录安全卡。

已具备的优点：信息架构清晰、agenda 统一数据源、reduced-motion 适配、回到顶部。下文每条改进均在此基线上做增量，不推翻现有结构。

## A. 易用性改进

### A1. "继续学习"智能直达（顶栏主按钮升级）

- **为什么改**：顶栏"进入课堂"主按钮目前只是页内锚点 `#dashboard-class-list`，点击后学生仍要在卡片网格里找课。学生 80% 的访问意图是"回到上次那门课"，当前路径多了一步扫读 + 一次点击。
- **怎么改**：
  1. 在 `_build_student_dashboard_context` 中计算"最近一次进入的课堂"（依据课堂访问记录或 `last_activity_at` 最近的 offering；若无记录则回退到第一个有待办的课堂，再回退到锚点行为）。
  2. `dashboard.html` 学生分支的 `navbar_actions` 按钮 `href` 改为 `/classroom/{id}`，副标题 `<small>` 显示课程名（截断 8 字），`title` 给全名。
  3. 无任何课堂时保持现状（锚点 + "继续学习"文案换成"查看课堂"）。
- **改成什么样**：学生登录后一次点击即回到上次学习的课堂；按钮副标题随上下文变化（如"继续 · 高级英语视听说"）。
- **验收标准**：
  - [ ] 有访问记录的学生：按钮直达对应 `/classroom/{id}`；
  - [ ] 新生无课堂：按钮回退为锚点且文案正确；
  - [ ] 课程名超长不撑破顶栏（CSS 截断）。
- **测试形式与预估结果**：P03 Playwright 测试，seed 学生访问某课堂后回首页，断言按钮 `href` 与副标题；预估一次通过（纯模板 + context 改动）。
- **偏差处理**：若没有现成的"课堂访问记录"表可查，**不要新建表**，降级为"`last_activity_at` 最新的课堂"即可满足 90% 场景；若副标题在窄屏顶栏放不下，去掉副标题只保留 `title` 提示，不牺牲顶栏布局。

### A2. 驾驶舱移动端折叠与优先级重排

- **为什么改**：cockpit 四个区块（提醒 / 统计 / 下一步 / 课程脉搏）在手机上垂直堆叠后首屏超过两屏半，"下一步"这个最高价值区块被挤到第二屏，违背"单一焦点"约定。
- **怎么改**：
  1. 在 `ui-system.src.css` polish 层为 `≤720px` 写媒体查询：cockpit 顺序重排为 **下一步 → 提醒 → 统计 → 课程脉搏**（grid `order` 或 `grid-template-areas`）。
  2. 统计四格在窄屏改为横向滑动单行（`overflow-x: auto` + scroll-snap），高度减半。
  3. "课程脉搏"在窄屏默认折叠，加"展开 N 门课程"切换按钮（小型自包含 JS，记忆到 `localStorage['lanshare:cockpit-pulse-collapsed']`）。
- **改成什么样**：手机首屏 = 问候 + 下一步前 2 条 + 提醒摘要，一屏内可决策"现在该做什么"。
- **验收标准**：
  - [ ] 390px 视口下"下一步"第一条在首屏可见；
  - [ ] 折叠状态跨刷新保持；
  - [ ] 桌面端布局零变化（媒体查询严格限定）。
- **测试形式与预估结果**：P03 harness `page.setViewportSize({width:390,height:844})` 截图 + `boundingBox` 断言首条 step 的 `y < 844`；预估首轮截图会发现间距问题，需 1 次微调。
- **偏差处理**：若 `order` 重排导致读屏顺序与视觉顺序不一致（无障碍问题），改用服务端输出两套区块顺序的方案代价过大，则保留 DOM 顺序、仅折叠次要区块来达成首屏目标。

### A3. 课堂搜索零结果引导 + `/` 快捷键聚焦

- **为什么改**：搜索无结果时只显示"没有匹配的课堂"，学生不知道是拼写问题还是筛选叠加导致；键盘用户没有快速聚焦搜索的方式。
- **怎么改**：
  1. `dashboard.js` 的空结果分支增强：显示当前生效的筛选/搜索词 chips，每个 chip 可单独点掉（复用现有 `data-reset-search` 逻辑拆分粒度）；并列出"接近的课堂"——用现有 `data-search-text` 做包含级别更宽的二次匹配（任意单字命中），最多给 3 个建议卡片链接。
  2. 全局 `/` 键（非输入态时）聚焦 `#dashboard-search`，`Esc` 清空并失焦；input 加 `placeholder` 提示"按 / 快速搜索"。
- **改成什么样**：零结果不再是死胡同，而是"哪里筛多了 + 你可能想找"的引导面板。
- **验收标准**：
  - [ ] 搜索"不存在课程X"时显示生效条件 chips 与 ≤3 条建议；
  - [ ] 点掉单个 chip 即时恢复对应结果；
  - [ ] `/` 聚焦、`Esc` 清空在 Chrome/移动端软键盘下不冲突（移动端不注册快捷键）。
- **测试形式与预估结果**：P03 键盘事件 + 断言 `document.activeElement`；零结果 DOM 断言。预估快捷键部分一次通过；模糊建议的匹配宽度需要 1–2 轮用 seed 数据调参。
- **偏差处理**：若宽松匹配产生大量噪音建议（中文单字命中过多），收紧为"双字连续命中"；仍不理想就砍掉建议卡片只保留 chips 引导——chips 是确定性收益，建议是锦上添花。

### A4. 新生首访引导（first-run 空状态升级）

- **为什么改**：无课堂学生看到的是通用空状态（标题 + 一个按钮），新生不知道"课堂从哪来"（等待教师拉入？自己加入？），开学周容易产生支持负担。
- **怎么改**：
  1. `dashboard_empty_state`（学生分支，`ui_copy_service.py` 中的文案）扩展为三步引导卡：① 确认个人信息（链 `/profile`）② 课堂由任课教师开设并导入名单，开课后自动出现 ③ 先去消息中心看通知（链 `/messages`）。
  2. 配一张轻量 SVG 插画（内联，使用 CSS 变量着色，适配主题）。
  3. cockpit 在零课堂时隐藏"课程脉搏"区块，避免空对空。
- **改成什么样**：新生首访 30 秒内理解平台的"课堂供给模型"，知道现在该做什么、不该焦虑什么。
- **验收标准**：
  - [ ] 零课堂学生账号渲染三步引导且链接可达；
  - [ ] 有课堂账号完全不受影响；
  - [ ] 引导文案在 `ui_copy_service.py` 而非模板硬编码。
- **测试形式与预估结果**：seed 一个零课堂学生跑 P03 截图 + 链接断言；预估一次通过。
- **偏差处理**：若三步内容与实际开课流程不符（例如未来支持学生自助加入），文案集中在 `ui_copy_service.py` 单点修改即可；插画若拖累美观（风格不搭）直接移除，引导价值在文字结构。

### A5. 时间感知问候与"今天的形状"

- **为什么改**：cockpit 标题/副标题是静态文案，缺少"现在是什么时候、今天还剩什么"的情境感；易用性上学生需要先建立时间锚点再决策。
- **怎么改**：`_build_student_cockpit` 注入时间感知字段：按服务器时间段生成问候（早上好/中午好/下午好/晚上好 + 姓名），副标题汇总"今天还有 N 节课 · M 项截止"（数据全部来自现有 `dashboard_agenda_events`，无新查询）；22:00 后若仍有未完成项，给一句克制的提示（"夜深了，剩余事项已为你保留到明天清单"）。
- **改成什么样**：标题像一位了解你今天日程的助教，而不是固定标语。
- **验收标准**：
  - [ ] 四个时间段问候正确（用注入时钟测试，避免真实时间依赖）；
  - [ ] 统计数字与 agenda 小组件一致（同源数据）；
  - [ ] 文案保持功能性、不卖萌（符合既有 copy 约定）。
- **测试形式与预估结果**：`_build_student_cockpit` 单元测试传入固定 `now`，四时段参数化断言；预估一次通过。
- **偏差处理**：若产品上认为晚间提示语越界（过于拟人），删掉该分支只保留问候 + 数字摘要；核心收益是时间锚点而非语气。

## B. 教学创新性改进

### B1. "AI 今日导学"卡片（接入平台既有 AI 体系）

- **为什么改**：平台已有 AI 服务（`ai_assistant.py`、agent-bridge 只读工具体系），但首页没有任何 AI 触点；学生的待办/作业/考试数据齐全，却要自己拼装"我今天该怎么安排"。这是平台最差异化的教学创新机会。
- **怎么改**：
  1. 新增服务端能力：每日为活跃学生生成一段 ≤120 字的"今日导学"——基于 agenda 事件 + 课程脉搏 + 错题摘要，由 AI 服务生成（**走既有统一调度器**注册一个每日清晨批量任务，缓存到表/缓存层，首页只读缓存，绝不在请求路径里同步调 AI）。
  2. cockpit "下一步"区块顶部插入导学卡：一句话学习建议 + 1 个"依据"标签（如"基于 3 项临近截止"），右上角"换一条"按钮限频（每天 ≤3 次，前端禁用态提示）。
  3. 卡片可整体关闭（`localStorage` 记忆 7 天），尊重不想要 AI 建议的学生。
  4. 降级链路：AI 不可用/未生成 → 卡片整体不渲染，页面无感知。
- **改成什么样**：学生每天打开首页先看到一句有依据、可解释的学习导引，例如"明天《数据结构》有作业截止，今晚优先完成第 3 题型的错题复盘（你上周错误率 40%）"。
- **验收标准**：
  - [ ] 首页请求路径零 AI 同步调用（响应时间不变）；
  - [ ] AI 故障/无缓存时卡片不渲染、无报错；
  - [ ] 建议文案包含至少一个可点击的落地链接（指向具体作业/材料）；
  - [ ] 关闭后 7 天内不再出现。
- **测试形式与预估结果**：服务层单测（mock AI 返回，断言缓存写入与降级）；P03 seed 缓存记录后截图断言卡片渲染与链接。预估服务端 2 轮（提示词与依据标签的拼装容易出格式问题），前端 1 轮。
- **偏差处理**：若生成质量不稳定（空话、与数据矛盾），第一阶段退化为**模板化规则文案**（不调 AI，纯规则拼装"最近截止 + 最弱课程"），保留卡片结构与数据管线，等提示词打磨好再切换 AI 源——卡片对用户的价值先于其智能程度。若 2c/4GB 服务器上批量任务造成负载尖峰，把批量改为"首次访问当天触发单人异步生成"。

### B2. 课程脉搏升级：七日学习节奏条 + 连续学习徽记

- **为什么改**：当前"课程脉搏"只有一个静态进度百分比，反映的是累计修为而非**学习节奏**；教学研究上"规律性"比"总量"更能预测成绩，学生也缺少坚持的正反馈。
- **怎么改**：
  1. `_build_student_cockpit` 为每门课增加近 7 日活跃位图（当日有提交/学习材料/进入课堂任一行为记 1；数据源用现有活动记录聚合，一条 SQL 按天 group）。
  2. 每张 pulse 卡进度条上方渲染 7 个小方格（CSS 即可，类似 GitHub 热力格，今天高亮描边），整体加 `aria-label="近7天学习 N 天"`。
  3. cockpit 统计区四格中替换价值最低的一格为"连续学习 N 天"（streak，跨课程取并集），配火苗/星标图标；中断不羞辱（归零只显示"今天开始新的连续记录"）。
- **改成什么样**：学生一眼看到每门课"这周我碰过几次"，节奏断档的课程方格灰白成片，自然形成行动暗示；连续学习徽记提供轻量游戏化激励。
- **验收标准**：
  - [ ] 7 日位图与活动数据一致（单测覆盖跨午夜/时区边界）；
  - [ ] streak 计算正确（含"今天尚未学习则显示昨天为止的 streak"规则）；
  - [ ] 纯 CSS 渲染，无新 JS 依赖；reduced-motion 下无动画。
- **测试形式与预估结果**：聚合函数单测（参数化日期边界）+ P03 截图核对方格渲染。预估单测 1 轮、SQL 聚合在 sqlite/postgres 双引擎下需验证日期函数兼容（`date()` vs `::date`），可能 1 轮返工。
- **偏差处理**：双引擎日期聚合若有坑，按既有模式走 `get_configured_db_engine()` 分支各写各的表达式；若 streak 被认为助长打卡式刷数据，把"进入课堂"从计分行为剔除，只算实质行为（提交、完成材料），口径集中在一个常量列表里便于调整。

### B3. 错题智能复习队列卡（间隔重复入口）

- **为什么改**：平台已有错题汇总（`assignment_wrong_summary`、cockpit 的 review plan item），但只是"下一步"里的一条链接，没有呈现**该复习什么、为什么是现在**；间隔重复是被验证最充分的学习科学手段，平台数据已支持却未产品化。
- **怎么改**：
  1. 服务端新增"今日复习队列"构建：从错题数据按"错误次数 × 距上次接触天数"打分排序，取 top 5，标注每条的复习理由（"3 天前错过 2 次"）。
  2. 右侧栏"优先处理"面板下方新增"今日复习"卡：题目摘要（截断）+ 课程名 + 理由 chip + 完成勾选（勾选调用现有 todo/复习标记接口或仅前端本日记忆，视后端现状取最小实现）。
  3. 全部完成时显示完成态插画 + "明天会有新的队列"。
- **改成什么样**：每天一份 5 题以内的轻量复习清单，有理由、可勾选、可清空，让"复习"从模糊任务变成确定动作。
- **验收标准**：
  - [ ] 排序分数函数单测覆盖（新错 > 旧错、多次错 > 单次错）；
  - [ ] 无错题学生不渲染该卡；
  - [ ] 勾选状态当日保持（刷新不丢）。
- **测试形式与预估结果**：打分函数单测 + P03 seed 错题数据截图。预估打分逻辑 1 次通过；勾选持久化方案取决于后端是否有现成标记位，若走 localStorage 则 1 轮，走后端接口加 1 轮。
- **偏差处理**：若错题数据结构不足以支撑"距上次接触"（没有复习行为记录），降级为"错误次数 + 题目时间"排序并把理由改为"错过 N 次"；不要为此新增行为埋点表，等复习行为有真实写入点后再升级口径。

### B4. "我的日程"时间轴模式补强为本周学习地图

- **为什么改**：学生端已有 `课程列表 / 我的日程` 切换，但时间轴模式只是把卡片按时间排序，与 agenda 小组件信息重复，模式价值不清晰，用户没理由切换。
- **怎么改**：
  1. 时间轴模式改造为**周视图学习地图**：横轴周一至周日，每天一列，列内按时间放置上课/作业截止/考试块（数据已在 `dashboard_agenda_events` + offering `timeline_items` 中），今天列高亮。
  2. 空白天显示极淡的"无安排"，整周空则提示切回列表。
  3. 块点击复用 agenda 既有的 detail popover（同一交互语言，不另起炉灶）。
  4. 切换状态记忆到 URL 参数（已有 `dashboard_default_group_mode` 管线）。
- **改成什么样**：切到"我的日程"看到的是一张本周课表式地图，回答"这周哪几天紧、哪几天松"，与 agenda 小组件的"接下来三件事"形成粗细两级互补。
- **验收标准**：
  - [ ] 七列渲染正确、今天列视觉突出；
  - [ ] 事件块颜色沿用既有 kind 配色（考试红 / 作业紫 / 上课青 / 待办蓝）；
  - [ ] 窄屏退化为"今天 + 明天"两列横滑；
  - [ ] popover 复用，无重复实现。
- **测试形式与预估结果**：P03 seed 跨周事件，截图 + DOM 列数断言 + popover 点击断言。预估布局 CSS 需 2 轮调（事件块重叠、长标题截断是常见坑）。
- **偏差处理**：若同一天事件过多导致列爆高，封顶显示 4 块 + "还有 N 项"折叠；若周视图实现成本超预期（重叠布局复杂），先交付"按天分组的垂直时间轴"（仍按天聚组、今天置顶展开），周地图作为二期。

## C. 美观性改进

### C1. 课堂卡片视觉身份：课程主题色与生成式纹理

- **为什么改**：课堂卡片目前全部同质（白底 + 同样的排版），学生靠读文字区分课程，扫读效率低；卡片网格作为页面最大面积区域，是美观性提升杠杆最大的地方。
- **怎么改**：
  1. 服务端按 `course_id` 稳定哈希到 8 个预设主题色（沿用既有 tone 体系：indigo/teal/sky/amber/rose/violet/emerald/slate），输出 `data-course-tone`。
  2. 卡片左上角加 4px 主题色顶边 + eyebrow（班级名）染同色系深色；卡片右上角铺一块低透明度（≤6%）的 SVG 几何纹理（按 course_id 选 4 种纹理之一：网格/波点/斜线/同心圆），CSS `mask` 实现，不引图片资源。
  3. "进入课堂"按钮保持全局主色不随卡片变色（保证主操作一致性）。
- **改成什么样**：每门课有稳定的"色彩+纹理"身份，学生形成肌肉记忆（"我的英语课是青色波点"），网格整体有韵律而不花哨。
- **验收标准**：
  - [ ] 同一课程在列表、脉搏卡、周地图中 tone 一致（同一哈希函数单点定义）；
  - [ ] 顶边色与白底文字对比度不受影响（纹理透明度 ≤6%，正文区域不铺纹理）；
  - [ ] 暗色文字对比度 ≥ 4.5:1（WCAG AA）。
- **测试形式与预估结果**：哈希函数单测（稳定性 + 分布）；P03 截图人工审美 check + 自动对比度抽查脚本。预估纹理透明度需 2 轮视觉调参。
- **偏差处理**：若纹理在截图评审中显脏（低分辨率屏上摩尔纹），去纹理只留顶边色——色彩身份是主收益，纹理可弃；若 8 色在课程多的学生页面出现相邻同色，接受之（哈希稳定性优先于相邻去重）。

### C2. 数字与进度的入场微动效

- **为什么改**：cockpit 统计数字、脉搏进度条、（新增的）七日方格目前是瞬时静态渲染，页面已有 `dashboard-reveal` 卡片级动效，但数据元素本身缺少"被点亮"的精致感；微动效是低成本高感知的美观投资。
- **怎么改**：
  1. 小型自包含模块 `static/js/dashboard_count_up.js`：`IntersectionObserver` 触发，数字从 0 滚动到目标值（400ms，easeOut），进度条宽度从 0 过渡到目标（CSS transition，JS 只负责加 `is-animated` 类）。
  2. 严格遵守 `prefers-reduced-motion: reduce` → 直接渲染最终值，模块提前 return。
  3. 只应用于首屏 cockpit 区（统计四格、脉搏进度、streak），不动右侧栏，避免满屏乱动。
  4. 记得给新 JS 加 `?v=` cache-buster。
- **改成什么样**：进入首页时核心数据有一次克制的"点亮"仪式感，0.4 秒内完成，不阻碍阅读。
- **验收标准**：
  - [ ] reduced-motion 下完全无动画且数值正确；
  - [ ] 动画结束值与服务端渲染值严格一致（动画失败兜底为静态值，即 HTML 初始就含真值、JS 只做视觉覆盖）；
  - [ ] 无 CLS（动画不改变元素尺寸）。
- **测试形式与预估结果**：P03 两次跑（默认 + `reducedMotion: 'reduce'` context）断言最终文本一致；`node --check` 语法检查。预估 1 轮通过，CLS 风险点在数字宽度变化（用 `font-variant-numeric: tabular-nums` 预防）。
- **偏差处理**：若低端设备掉帧（2c 服务器不影响前端，但学生设备可能差），把数字滚动砍掉只留进度条 CSS transition；任何动画问题的兜底永远是"删动画"，因为 HTML 初始值即真值，删除零风险。

### C3. 右侧栏视觉减重与分组节奏

- **为什么改**：右侧栏目前四个面板（快捷入口/优先处理/最近动态/登录安全）每个都是"标题 + 副标题 + 列表"的同等视觉重量，副栏与主列争夺注意力，违背"副栏弱化"约定；登录安全卡五行键值对的信息价值低但占位高。
- **怎么改**：
  1. 副栏面板统一降级样式：面板标题字号降一级、去副标题（副标题信息并入 `title` 提示）、面板间距收紧、背景由实卡改为更轻的描边卡。
  2. 登录安全卡折叠为单行摘要（"最近登录：6-11 22:03 · Windows · 密码登录"）+ 点击展开详情（`<details>` 原生元素，零 JS）；异常信号（新设备/新 IP）时自动展开并加 amber 边。
  3. "最近动态"超过 5 条折叠（与既有"过往项折叠"约定一致）。
- **改成什么样**：右侧栏成为安静的工具带，主视线自然落在 cockpit 与课堂列表；安全信息平时一行、异常时自己跳出来。
- **验收标准**：
  - [ ] 右侧栏总高度较现状减少 ≥30%（seed 标准数据下量测）；
  - [ ] 安全卡 `<details>` 键盘可操作（Enter/Space）；
  - [ ] 新设备场景自动展开逻辑有服务端字段支撑并被单测覆盖。
- **测试形式与预估结果**：P03 `boundingBox` 高度对比 + 截图评审；展开交互断言。预估 1–2 轮（高度目标可能需再收间距）。
- **偏差处理**：若"新设备判定"现有数据不可靠（device_label 缺失率高），先不做自动展开，只做折叠——折叠是确定收益；30% 高度目标若达不到，接受 20% 但必须保住"标题降级 + 安全卡折叠"两个结构性改动。

### C4. 全页视觉基调统一：间距尺、圆角与阴影 token 审计

- **为什么改**：首页由多轮迭代叠加而成（cockpit、agenda、卡片、日历来自不同时期），间距（12/14/16/18/20px 混用）、圆角（10/12/14/16px 混用）、阴影深浅不一，整体"接近精致但差一口气"；美观性的最后 20% 来自一致性而非新元素。
- **怎么改**：
  1. 在 `ui-system.src.css` polish 层定义首页作用域 token：`--dash-gap-{sm,md,lg}`、`--dash-radius-{card,chip}`、`--dash-shadow-{rest,hover}`（取现状中位数为基准值，限 `body.dashboard-page` 作用域，不碰其他页面）。
  2. 一次性把首页各区块的硬编码值替换为 token（纯查找替换式重构，不改布局结构）。
  3. hover 阴影统一为单一规格 + `translateY(-2px)`，去掉个别卡片的夸张抬升。
- **改成什么样**：所有卡片同一圆角、同一静息/悬停阴影、间距呈 8 的倍数节奏，页面"安静下来"。
- **验收标准**：
  - [ ] 首页范围内 `border-radius` / `box-shadow` 字面值替换为 token（grep 首页相关 class 块验证）；
  - [ ] 改动前后逐区块截图 diff，无布局位移（仅圆角/阴影/间距微差）；
  - [ ] 其他页面（教师端、manage、课堂页）截图零变化。
- **测试形式与预估结果**：P03 全页 + 分区块截图 before/after 对比（含一张教师端对照截图防泄漏）；预估 2 轮（总会漏几个角落值）。
- **偏差处理**：若某处替换后视觉明显变差（个别组件依赖原值），允许该组件保留局部覆盖值并加注释说明例外；若 token 化波及面失控（选择器作用域穿透到课堂页），立即收窄选择器而不是扩大 token 适用范围——本条的纪律是"只统一、不重设计"。

## D. 实施顺序与依赖

| 批次 | 内容 | 理由 |
|------|------|------|
| 第 1 批 | A1、A5、C3、C4 | 纯 context/模板/CSS，低风险高感知，先把"底色"做对 |
| 第 2 批 | A2、A3、A4、C2 | 交互与移动端，依赖第 1 批的视觉基线 |
| 第 3 批 | B2、C1 | 需要新数据聚合（七日活跃）与 tone 体系打通 |
| 第 4 批 | B3、B4 | 复习队列与周地图，工作量最大 |
| 第 5 批 | B1 | AI 导学依赖调度器批量任务与缓存管线，且需提示词打磨期 |

每批独立可交付、独立可回滚（feature branch + 部署后 `curl` 验证静态资产 + 截图复核，按既有 deploy-workflow 执行）。

## E. 统一测试与验收流程

每条改进遵循同一闭环（即 frontend-change-playbook）：

1. **改前基线**：P03 harness 以 seed 学生账号截图现状（桌面 1280 + 移动 390 两档）。
2. **静态检查**：`node --check` 改动的 JS、Jinja parse 模板、`npm run build`、`npx tsc --noEmit`。
3. **行为断言**：每条的验收标准转为 Playwright 断言（DOM/href/viewport/键盘），服务端逻辑配 pytest 单测（时间、打分、哈希函数必须参数化测试）。
4. **无障碍抽查**：键盘可达、`aria-label`、对比度（C1/C3 重点）。
5. **控制台零报错**：`collectBrowserErrors` 必须为空。
6. **部署验证**：deploy 后 `curl -sk` 确认带新 `?v=` 的资产可达，线上再截一轮图。

**整体预估结果**：A 组与 C 组共 9 条中预计 7 条一轮过、2 条需视觉/布局二轮调参；B 组 4 条均涉及服务端，预计每条 1–2 轮返工，B1 风险最高（AI 质量不可控，已内置规则文案降级路径）。

**全局偏差原则**：任何一条若实施中发现成本/风险超出预估，按以下优先级裁剪——
1. 先砍"智能/生成"部分保留"结构/数据"部分（B1、B3 已写明降级形态）；
2. 再砍动效与装饰保留信息与交互（C1 纹理、C2 数字滚动可弃）；
3. 结构性收益（A1 直达、A2 首屏重排、C3 减重、C4 token）不可裁剪，只可分批；
4. 所有降级决策记录回本文档对应条目，保持文档与实现同步。
