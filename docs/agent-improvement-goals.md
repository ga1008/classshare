# LanShare Agent 功能改进执行目标

> 版本：v1.0（2026-06-11）
> 前置状态：Agent 任务中心已具备「全平台单任务队列 + deepseek-tui 独立运行时 + agent-bridge 只读桥接（SQL/文件/联网）+ 平台全知提示词注入」（commit 3116b15）。
> 总方向：**一切朝易用性努力** —— 教师"看得见、追得上、用得上、不用等、不用叫"。

## 总目标

把 Agent 从"提交后黑盒等待的顾问"升级为"全程可见、可对话、结果可一键落地、能定时主动工作的平台助手"。

## 核心原则（每条改进都必须遵守）

1. **只读边界不破**：Agent 对平台数据与代码永远只读；任何写入必须经过平台白名单动作 + 教师显式确认。
2. **降级优先**：任何新链路失败时回退到现有行为（轮询快照、静态卡片、人工操作），不能比现在差。
3. **资源现实**：生产是 2 核 / 4GB VPS，约 200 并发用户。新增常驻开销（连接、线程、轮询频率）必须给出预算并实测。
4. **验收以教师视角**：每条验收标准都写成"教师能做到什么"，而不是"接口返回什么"。

## 实施优先级总览

| 阶段 | 条目 | 主题 | 解决的问题 |
|------|------|------|-----------|
| P1（先做） | G1 / G3 / G6 | 实时过程流、结果落地按钮、定时 Agent 任务 | 看不见、用不上、不主动 |
| P2 | G2 / G4 / G5 | 任务可对话、附件进任务、队列体验 | 追不上、喂不进、等不起 |
| P3 | G7 / G8 / G9 / G10 | 桥接增强、任务记忆、聊天工具循环、失败自愈 | 查不准、记不住、杀鸡用牛刀、一错到底 |

## G1 实时过程流（任务执行透明化）

### 为什么改
当前 `agent_task_worker` 每 5 秒轮询 deepseek-tui 的任务快照写入 DB，前端再每 5 秒轮询 `/api/agent-tasks`，教师看到的是一张**静态卡片**，任务跑 3 分钟里教师不知道 Agent 在干什么、卡没卡死。"黑盒等待"是目前最大的体验杀手——用户会反复点刷新、甚至误以为失败而取消。

### 怎么改
1. **后端事件化**：worker 在轮询运行时快照时做 diff，把 `timeline` / `tool_calls` 的**增量**写入现有 `agent_task_events` 表（新增 event_type：`runtime_step`、`runtime_tool_call`、`runtime_output_delta`），事件 detail 里带人话化描述（"正在查询数据库：courses 表"、"正在抓取网页：example.com"）。
2. **推送通道**：新增 `GET /api/agent-tasks/{id}/stream`（SSE）。实现上复用主应用现有的异步能力（chat WebSocket 已证明事件循环可承载），SSE 内部以 1~2 秒间隔从 `agent_task_events` 拉增量推给前端；不引入新进程间通信。
3. **前端流式渲染**：Agent 任务卡片增加"执行过程"区，事件逐条淡入，类似聊天打字流；运行中显示当前步骤 + 已运行时长；SSE 不可用（代理/旧浏览器）时自动回退现有 5 秒轮询。
4. **人话化映射表**：工具调用名 → 中文动作描述维护在 `agent_task_service`，未知工具名显示原始名，不阻塞。

### 改成什么样
教师提交任务后，卡片实时滚动显示："已理解任务 → 正在查询课堂数据（第 2 次查询）→ 正在生成草稿 → 完成"。中途任何时刻打开页面都能看到当前进度和过程历史。

### 验收标准
- 提交一个会触发 ≥3 次工具调用的任务，教师在卡片上能看到 ≥3 条过程事件，且事件出现延迟 ≤3 秒（相对 worker 写库时间）。
- 断开 SSE（开发者工具断网再恢复）后卡片自动恢复更新，无白屏、无重复事件。
- 任务页关闭重开，过程历史完整可见（事件持久化在 DB）。
- 主应用在 20 个并发 SSE 连接下，`/api/internal/health` 响应时间无可见劣化（<200ms 增量）。

### 测试形式与预估结果
- **单测**：快照 diff → 事件生成的纯函数测试（给定两个 runtime 快照，断言产出事件序列）。预估：全绿，diff 逻辑约 5~8 个用例。
- **集成**：P03 harness（SQLite）+ 模拟 runtime 快照序列，断言 events 表内容与 SSE 输出一致。
- **E2E**：Playwright 登录教师 → 提交任务（runtime 用 stub 返回预置 timeline）→ 断言卡片出现过程文本。预估：一次通过；最可能的偏差是 SSE 在 uvicorn 单 worker 下与长轮询争抢线程。
- **负载**：`full_stack_load_test.py` 加 20 路 SSE 场景，观察事件循环延迟。

### 偏差处理（朝易用性）
- 若 SSE 在单 worker 下造成事件循环压力 → 降为 2 秒间隔的**短轮询增量接口**（`?after_event_id=`），前端体验几乎等价，先保流畅再谈实时。
- 若 runtime 快照里 timeline 字段不稳定（deepseek-tui 版本差异）→ 仅展示状态机级别事件（排队/思考/工具调用计数/完成），宁可粗粒度也不展示错误信息。
- 若人话化映射覆盖率低 → 默认隐藏原始工具名，只显示"正在执行第 N 步"，避免暴露技术细节吓到非技术教师。

## G2 任务可对话（追问 / 补充 / 中途纠偏）

### 为什么改
任务现在是一锤子买卖：结果不满意只能新建任务、重新排队、重新提供上下文。deepseek-tui 原生有 `thread_id`（已存在 `agent_tasks.runtime_thread_id`），续聊能力是现成的，浪费了。

### 怎么改
1. 任务终态卡片（completed/failed）增加输入框："对这个结果继续提要求"。
2. 新增 `POST /api/agent-tasks/{id}/follow-up`：校验任务归属 + 终态 + thread 存在，创建一条**子任务**（新 task 行，`parent_task_id` 指向原任务，复用原 context + 新指令），入队时 worker 对 runtime 用同一 `thread_id` 续聊（`/v1/tasks` 带 thread 参数；若 runtime 不支持则把"上次任务摘要 + 上次结论"拼进新 prompt——效果等价）。
3. 运行中任务追加"补充说明"：写入一条 `pending_supplement` 事件，worker 在下一次轮询时若 runtime 支持注入则注入，不支持则在任务完成后提示"补充说明将作为追问发送"。
4. 前端把父子任务在卡片里串成一条会话链（缩进或时间线样式），删除父任务时级联提示。

### 改成什么样
教师看到结果后直接在卡片里打字"第二条再展开些，加上学时要求"，Agent 带着完整上文继续，不重新排长队（追问任务在队列里优先级提升一档）。

### 验收标准
- 对已完成任务发追问，新任务自动携带原任务上下文，结果明显延续上文（人工评审 5 例，≥4 例不需要教师重复背景）。
- 追问任务从提交到开始执行的等待时间 ≤ 新任务的 50%（优先级验证）。
- 父子任务在 UI 上呈链式展示，互不串扰；删除整链一次确认完成。
- 对他人任务 / 进行中任务发追问被拒绝（403/409），提示文案友好。

### 测试形式与预估结果
- **单测**：follow-up 创建逻辑（归属/状态校验、parent 链、优先级标记）约 6 个用例，预估全绿。
- **集成**：stub runtime 断言续聊请求带 thread_id 或 prompt 含上次摘要。
- **E2E**：Playwright 提交→完成（stub）→追问→断言链式卡片。预估：UI 链式渲染需要 1~2 轮样式调整。
- **人工评审**：5 组真实追问对话质量打分。

### 偏差处理
- 若 deepseek-tui 的 thread 续聊行为不可靠（上下文丢失/串线）→ 永久走"摘要拼接"方案：把父任务 instruction + result_summary + 关键产物路径拼进新 prompt。易用性不受影响，只是 token 多一点。
- 若优先级插队引发普通任务饥饿 → 改为"追问只比同教师的新任务优先"，全局仍按 FIFO。
- 若运行中注入补充说明导致 runtime 行为异常 → 砍掉运行中注入，只保留终态追问（覆盖 90% 场景）。

## G3 结果落地按钮（草案 → 一键执行）

### 为什么改
Agent 只能输出"草案 + 建议"，教师要自己复制内容、找到对应页面、手动建作业/发通知/存材料。"最后一公里"全是手工，Agent 的产出价值打了对折。这是把 Agent 从顾问变成助手的关键一步。

### 怎么改
1. **结构化动作提案协议**：在 runtime prompt 中约定，凡涉及平台状态变更，Agent 在最终输出里附带 `proposed_actions` JSON 块（action 类型 + 完整参数 + 人话摘要）。worker 解析后存入 `result_detail.proposed_actions`。
2. **白名单动作注册表**：新建 `agent_action_registry`（复用/扩展现有 `agent_platform_actions.py`），首批 4 个动作：
   - `create_assignment_draft`（建作业，状态=草稿不发布）
   - `send_student_notification`（站内通知，先预览收件人名单）
   - `save_material_draft`(存为课堂材料，标记 AI 生成)
   - `create_blog_draft`（博客草稿箱）
   每个动作定义：参数 schema 校验、执行函数、回滚函数、风险等级。
3. **确认执行 UI**：任务卡片渲染动作提案为按钮组，点击弹出参数预览（可编辑标题/收件人等关键字段）→ 教师确认 → `POST /api/agent-tasks/{id}/actions/{action_index}/execute` → 平台以**教师身份**执行白名单函数 → 卡片显示执行结果 + 跳转链接。
4. **审计**：每次执行写 `agent_task_events`（谁、何时、什么动作、参数摘要、结果），高风险动作（发通知）执行后 10 分钟内可撤回（通知未读则撤回）。

### 改成什么样
Agent 答完"已为你拟好作业草案"，下面直接出现【创建为作业草稿】按钮；点开预览、改个截止日期、确认——作业出现在课堂里（草稿态），按钮变成"已创建 → 去查看"。

### 验收标准
- 4 个白名单动作全部走通：提案 → 预览 → 编辑 → 执行 → 跳转查看，全程不离开 AI 助手窗口。
- 未经确认绝不写入：断言执行接口在无确认 token / 非任务归属者 / 参数被篡改超出 schema 时拒绝。
- 动作执行失败时卡片显示具体原因和"去手动操作"的兜底链接（永远有人工出路）。
- 审计事件完整：每次执行可追溯到教师、时间、参数。
- 模型输出的 `proposed_actions` 解析成功率 ≥80%（解析失败时降级为普通文本展示，不报错）。

### 测试形式与预估结果
- **单测**：动作 schema 校验、提案 JSON 解析（含残缺/恶意 JSON）约 12 个用例。预估全绿；恶意参数用例可能暴露 1~2 个 schema 漏洞，属预期收获。
- **集成**：P03 harness 全链路——stub runtime 返回含提案的结果 → API 执行 → 断言 DB 中作业草稿存在且归属正确。
- **E2E**：Playwright 点击按钮流程 + 截图。预估：预览弹窗的表单交互需要 2~3 轮打磨。
- **提示词回归**：10 个典型任务跑真实模型，统计提案 JSON 合规率。预估首轮 60~70%，调提示词（给 JSON 模板 + few-shot）后达到 80%+。

### 偏差处理
- 若模型 JSON 合规率始终 <80% → 改用"二段式"：任务完成后由平台再发一次廉价的结构化抽取请求（固定 schema 的 function calling）从自然语言结果中抽提案。多一次 AI 调用换稳定结构，值得。
- 若某动作执行路径耦合太深（如建作业牵扯发布流程）→ 该动作先降级为"预填表单跳转"：按钮直接跳到对应管理页并把参数填好，教师点保存。仍然省去手工录入，易用性保住大半。
- 若教师误触担忧大 → 所有动作首批一律落草稿态（不直接面向学生），观察一个月再考虑放开"直接发布"。

## G4 附件进 Agent 任务

### 为什么改
聊天模式支持传图片/文档，Agent 模式当前直接丢弃附件并提示"暂不携带附件"。教师最自然的需求"帮我按这份名单/这页截图处理"被挡在门外，而成本极低——任务 workspace 本来就是个目录。

### 怎么改
1. 前端：Agent 模式不再禁用附件按钮；提交时附件随 FormData 上传到 `POST /api/agent-tasks`（multipart 化现有接口，保持 JSON 字段兼容）。
2. 后端：复用聊天的 `_process_chat_file` 解析（文本类抽全文、docx/pdf 抽文本、图片存原图），写入任务 workspace `attachments/` 子目录；`TASK.md` 中列出附件清单与摘要。
3. runtime prompt 增加附件段落："教师提供了 N 个附件，路径与摘要如下，可直接读取"。图片附件在创建任务时先用多模态模型生成一段文字描述写进上下文（runtime 本身若不支持视觉，文字描述兜底）。
4. 限制：单任务 ≤5 个文件、单文件 ≤10MB、总量 ≤20MB；超限前端即时提示。

### 改成什么样
教师拖一份 Excel 名单 + 一句"按这份名单生成分组讨论安排"，Agent 直接读文件干活，产物里引用名单中的真实姓名。

### 验收标准
- txt/md/docx/xlsx/pdf/图片 六类附件各提交一次，Agent 的输出能证明确实读到了内容（引用文件中的具体信息）。
- 附件超限时提交被前端拦截，提示具体哪个文件超限。
- 附件文件落在该任务的隔离 workspace 内，任务删除时一并清理。
- 聊天模式行为零回归（附件仍走原聊天链路）。

### 测试形式与预估结果
- **单测**：附件落盘路径、清单生成、超限校验约 8 个用例。预估全绿。
- **集成**：上传 xlsx → 断言 workspace 中存在解析文本 + TASK.md 含清单。
- **E2E**：Playwright 上传文件提交任务，断言无报错提示。预估一次通过。
- **真实任务**：6 类文件各跑一个真实任务人工验证引用质量。预估 xlsx/pdf 的表格类内容抽取质量一般（现有解析器限制）。

### 偏差处理
- 若 xlsx/pdf 表格抽取质量差 → 在附件摘要里如实标注"表格类文件，建议教师在指令中说明关键列含义"，同时把原文件路径给 Agent（runtime 有 shell，可自行用 python 读）——把解析智能交给 Agent 而不是平台硬编码。
- 若图片描述生成拖慢任务创建（多模态调用 2~5 秒）→ 改为异步：任务先入队，描述生成完成后追加到 workspace；worker 取任务时若描述未就绪等待 ≤10 秒。
- 若磁盘占用增长快（70GB SSD）→ 任务终态 7 天后自动清理附件目录（保留文本摘要），纳入现有 scheduler。

## G5 队列体验（等待可预期、完成必通知）

### 为什么改
全平台单任务队列：多教师同时用时排队不可避免，但当前排队者只能看到"队列第 N"，不知道等多久；任务完成时若教师已关页面，无任何通知，常常半小时后才想起来看。**等待本身可以接受，不可预期的等待不可接受。**

### 怎么改
1. **预估等待**：基于最近 20 个完成任务的耗时中位数 × 前方排队数，卡片显示"预计还需 ~N 分钟"（粗粒度：<5 分 / 5-15 分 / >15 分三档措辞）。
2. **完成通知**：任务终态时写消息中心通知（复用 `message_center_notifications`）+ 浏览器内 toast（若页面开着）；教师在邮件提醒设置中可选开启邮件通知（复用 email_outbox + scheduler 一次性任务模式）。
3. **并发提升**：deepseek-runtime `--workers` 从 1 提到 2，配套把"全平台单任务"约束放宽为"每教师同时 1 个 + 全局同时 2 个"（agent_tasks 的 claim SQL 已支持 advisory lock，改并发上限即可）。Agent 任务大部分时间在等模型 API 响应，不吃 CPU，2 核可承受。
4. **队列公平**：同一教师连续提交多个任务时，第 2 个起排在其他教师首个任务之后（防独占）。

### 改成什么样
排队时看到"前面 2 个任务，预计 ~8 分钟，完成后会在消息中心通知你"，教师可以放心去干别的。

### 验收标准
- 排队卡片显示预估等待，且实际完成时间落在预估档位内的比例 ≥70%（统计 20 个任务）。
- 关闭页面后任务完成，消息中心铃铛出现通知，点击直达任务卡片。
- 两个不同教师的任务可同时运行；同一教师第二个任务正确排队。
- VPS 负载：两任务并发时 load average <2.0、内存余量 >500MB（实测）。

### 测试形式与预估结果
- **单测**：预估算法、公平排序 SQL 约 6 个用例。预估全绿。
- **集成**：双教师 fixture 并发提交，断言同时 running=2、同教师串行。
- **负载实测**：生产镜像本地 compose 起双任务跑 30 分钟观察资源。预估内存是瓶颈而非 CPU（deepseek-tui 每 worker 常驻 ~200-400MB）。
- **E2E**：通知铃铛断言。

### 偏差处理
- 若双 worker 内存超预算 → 回退 `--workers 1`，只保留"预估等待 + 完成通知"（这两项已解决 80% 的等待焦虑，是本条的易用性底线）。
- 若预估经常不准（模型 API 波动）→ 把档位措辞改得更模糊（"很快 / 需要一会儿 / 队列较长"），宁可模糊不可误导。
- 若消息中心通知被教师反映打扰 → 默认只通知"自己主动关闭页面前未看到结果"的任务（卡片可见时完成则不另发）。

## G6 定时 / 订阅型 Agent 任务（主动的灵魂）

### 为什么改
平台已有通用 scheduler（专用 worker 容器 + handler 注册表），而 Agent 只能被动等指令。"每周一汇总上周作业情况""新公文提到我时自动解读"这类周期性/事件触发需求，正是"时刻待命的灵魂"的核心定位，目前完全缺失。

### 怎么改
1. **scheduler 接入**：注册新 handler `agent_task_dispatch`：到点把预设的 Agent 任务模板（instruction + context 构造器）写入 agent_tasks 队列，复用全部现有执行链路。
2. **订阅模板首批 3 个**（预置而非自由编排，降低理解成本）：
   - 周报：每周一 08:00 汇总该教师上周作业提交率、低分预警、课堂活跃（查库统计 + AI 解读）。
   - 公文哨兵：每天 09:00 检查 gongwen_documents 新增中命中教师关注关键词/姓名的文档，生成解读。
   - 考前提醒包：考试前 3 天自动生成监考安排核对 + 学生注意事项草稿。
3. **订阅管理 UI**：AI 助手窗口新增"定时任务"入口（Agent 模式下的次级标签）：开关每个模板、设定时间、查看历史产出列表。底层用现有 scheduled_tasks 表（cron 表达式 + handler 参数 JSON）。
4. **产出送达**：定时任务产出写任务历史 + 消息中心通知（标题如"📋 本周教学周报已生成"）；连续 2 次产出"无新内容"时自动降频提示（"本订阅最近没有新内容，是否暂停？"）。

### 改成什么样
教师周一早上打开平台，铃铛里躺着一份 AI 周报："上周 3 个班作业提交率 92%/88%/95%，王某某连续两次未交（建议关注），本周四有监考……"——没有任何人触发过它。

### 验收标准
- 3 个模板均可订阅/退订/改时间，配置即时生效（下个周期按新配置跑）。
- 到点 ±2 分钟内任务入队（scheduler 现有精度），产出出现在任务历史 + 消息中心。
- 周报内容数据准确：抽 1 期人工核对提交率数字与管理页一致。
- 订阅任务与手动任务在队列中互不阻塞超过一个任务时长（定时任务优先级低于手动任务）。
- 退订后不再产生任务（观察 2 个周期）。

### 测试形式与预估结果
- **单测**：handler 参数构造、cron 生效、降频判断约 8 个用例。预估全绿。
- **集成**：把 cron 设为 1 分钟后，断言任务入队、产出落库、通知生成（P03 harness + stub runtime）。
- **真实数据演练**：用生产数据副本跑一期周报人工评分。预估首版解读质量中等，需要 1~2 轮 prompt 调优（数字部分由 SQL 保证准确，AI 只做解读，风险可控）。
- **长跑**：staging 连续跑 2 周观察稳定性与降频逻辑。

### 偏差处理
- 若定时任务挤占手动队列引发抱怨 → 定时任务限定在低峰时段执行（06:00-08:00），且全局并发为 1 时手动任务可抢占（定时任务重新排队）。
- 若 AI 解读出现数字编造 → 周报改为"SQL 统计表格（平台生成，绝对准确）+ AI 仅写一段建议"，结构上隔离事实与生成。
- 若教师觉得通知吵 → 所有订阅默认关闭，首次进入 Agent 模式时用一条引导卡片介绍（选择加入，而非默认打开）。

## G7 桥接接口增强（让 Agent 查得准）

### 为什么改
实测中模型写 SQL 的主要失败模式：引号转义错、列名记错、把检索写成复杂 LIKE 拼接。每次失败 = 一次工具调用浪费 = 任务变慢变差。桥接接口应该把"容易写错的部分"产品化。

### 怎么改
1. `POST /query` 支持参数化：`{"sql": "SELECT ... WHERE name = :name", "params": {"name": "张三"}}`（sqlite/psycopg 双引擎适配已有基础设施 `sqlite_sql_to_psycopg`）。
2. 新增 `POST /search`：统一关键词检索端点，`{"scope": "gongwen|materials|assignments|all", "keyword": "...", "limit": 20}`，内部用各表既有的检索逻辑（公文检索服务已存在），返回统一结构（type/title/snippet/url/date）。
3. `POST /file` 支持 docx/pdf：复用聊天链路的文本抽取器，返回纯文本。
4. `GET /meta` 增加 `example_queries`：按任务类型给 5~8 条**实测可跑**的示例 SQL（含正确的表名列名 join 方式），并标注时间列格式约定。
5. BRIDGE.md 同步更新所有新端点的 curl 示例。

### 改成什么样
Agent 检索公文从"自己拼三层 LIKE"变成一次 `/search` 调用；查统计时照着 example_queries 改参数，首次成功率显著上升。

### 验收标准
- 参数化查询双引擎（sqlite 本地 / postgres 生产）均正确执行与转义。
- `/search` 三个 scope 各自返回正确结果且带可点击站内 url。
- docx/pdf 经 `/file` 返回的文本与聊天附件解析结果一致。
- 用 10 个典型任务实测：Agent 工具调用的 SQL 首次成功率从基线（先测，预估 ~60%）提升到 ≥85%。

### 测试形式与预估结果
- **单测**:参数化转义（注入向量用例 ≥6 个）、search 各 scope、file 类型分支，约 15 个用例。预估全绿；注入用例可能暴露转义边角，属预期收获。
- **集成**：postgres 容器跑同套查询断言双引擎一致。
- **基线对比实验**：同 10 个任务在改进前后各跑一遍，统计工具调用成功率与任务总时长。预估任务平均耗时下降 20~30%。

### 偏差处理
- 若参数化在 `sqlite_sql_to_psycopg` 适配中出兼容问题 → 保留常量 SQL 路径不动，参数化作为增量能力（文档里推荐但不强制），不阻塞发布。
- 若 `/search` 各表检索质量参差 → 先只放 gongwen（已有成熟检索服务），materials/assignments 标注 beta，按反馈逐个打磨。
- 若示例 SQL 随 schema 演进过期 → 给 example_queries 加 CI 冒烟测试（每条示例在测试库上必须可执行），从机制上防腐。

## G8 任务记忆（Agent 记得这位教师）

### 为什么改
每个任务都是冷启动：Agent 不知道教师上次让它生成过什么风格的文档、做过什么决定。教师被迫重复交代偏好（"还是上次那种格式"是说不通的）。

### 怎么改
1. **轻量方案（本期）**：`build_runtime_prompt` 注入该教师最近 5 条已完成任务的「标题 + 一句话结论摘要」+ 最近 1 条同类型任务的产物路径（Agent 可经 `/file` 读上次产物参考风格）。
2. 摘要来源：任务完成时已有 `result_summary`，无需新增生成成本；截断到每条 ≤100 字。
3. 教师可控：任务历史抽屉加"不带入历史"开关（敏感任务勾选后不进记忆窗口，且本次也不读历史）。

### 改成什么样
教师说"再生成下一课的导学文档"，Agent 自动参考上次导学文档的结构和风格，产出连贯一致。

### 验收标准
- runtime prompt 中可见历史块（≤600 字预算，超出截断最旧）。
- 同类型任务连续做 2 次，第 2 次产物结构与第 1 次一致性明显（人工评审 3 组，≥2 组认可）。
- 勾选"不带入历史"后 prompt 无历史块（日志断言）。
- 历史块不含其他教师任何信息（隔离断言）。

### 测试形式与预估结果
- **单测**：历史拼装、截断、开关、隔离，约 6 个用例。预估全绿。
- **人工评审**：3 组连续任务对比。预估风格延续性提升明显，但要警惕"错误也被延续"。

### 偏差处理
- 若历史造成"错误风格固化"（教师其实想换风格）→ 历史块措辞改为"仅供参考，教师本次指令优先"，并在 UI 提交框 placeholder 提示"想换风格请直接说明"。
- 若 600 字预算挤占主上下文 → 降到只注入最近 1 条同类型任务，效果损失最小。
- 长期方案（不在本期）：教师级偏好档案（喜欢的文档结构/语气）由 AI 从任务历史定期蒸馏——留到 G6 定时任务体系成熟后做。

## G9 聊天 AI 工具循环（轻问题不动用 Agent 队列）

### 为什么改
"帮我查一下三班有多少人没交作业"是轻量查询，现在却只有两条路：聊天 AI 靠预注入画像猜（可能没有该数据），或动用 Agent 队列（杀鸡用牛刀且要排队）。聊天 AI 缺一轮实时查库能力。

### 怎么改
1. AI 侧（ai_assistant.py）：对支持 function calling 的模型（deepseek/volcengine 均支持）注册 1 个工具 `platform_query`（参数即 agent-bridge `/query` 的子集），最多允许 2 轮调用，超出则提示"建议转 Agent 任务"。
2. 鉴权差异：聊天工具调用以**当前用户身份**走内部函数（不经 HTTP 桥接），查询范围比 Agent 更窄——教师限自己课堂相关表的预定义视图（白名单视图列表），学生不开放。
3. 流式协议扩展：新增 `tool_status` 事件（"正在查询平台数据…"），前端聊天气泡显示工具调用状态徽标，结果出来后正常流式作答。
4. 转化引导：当 2 轮工具不够用时，AI 回复末尾自动附"这个问题适合交给 Agent 任务深入处理 →"快捷按钮（一键把当前问题转为 Agent 任务）。

### 改成什么样
教师在任何页面问"三班几个人没交本周作业"，3 秒内得到准确数字和名单，不排队、不开任务。

### 验收标准
- 5 个典型轻查询（提交率/人数/某生成绩/公文检索/日程）均一次往返答对（与管理页数字核对）。
- 工具调用期间气泡显示状态，无"假死"观感。
- 学生账号提问不触发工具（断言不发 function 定义）。
- 工具失败（SQL 错/超时）时 AI 用自然语言说明并给出人工查询路径，不裸抛错误。
- 单轮对话 P95 延迟 ≤8 秒（含一次工具调用）。

### 测试形式与预估结果
- **单测**：白名单视图鉴权、轮数限制、事件编码约 8 个用例。预估全绿。
- **集成**：mock 模型返回 tool_call，断言查库与二次请求拼装正确。
- **真实模型评测**：20 个轻查询统计答对率。预估首轮 70~80%（模型选错视图是主要失败源），把视图描述写细后可到 85%+。
- **E2E**：聊天气泡状态徽标断言。

### 偏差处理
- 若模型工具调用准确率低 → 缩减视图数量到 5 个最高频（提交情况/班级名册/公文检索/我的日程/课堂列表），少而准胜过多而乱。
- 若延迟超标 → 工具调用模型固定用 fast 档（已有 task_type 路由机制），思考留给最终作答。
- 若与 Agent 边界混淆（用户在聊天里要求重活）→ 强化转化引导文案，并在轮数耗尽时**自动**带上下文预填 Agent 任务输入框，一键提交。

## G10 失败自愈（一错不到底）

### 为什么改
runtime 失败现在直接 failed 终态，哪怕只是模型 API 瞬时 503 或输出超长被截断。教师视角：白等十分钟得到一句"任务失败"，所有上下文作废。失败任务占比（生产已有 1 例 failed）每一例都是强烈负面体验。

### 怎么改
1. **错误分类**：worker 把 runtime 错误归类为 `transient`（网络/限流/超时）/ `content`（输出解析失败）/ `fatal`（指令违规/runtime 崩溃）。
2. **自动重试**：transient 类自动重试 1 次（间隔 30 秒，事件流显示"遇到临时故障，正在自动重试"）；content 类重试时在 prompt 附加上次失败原因（"上次输出 JSON 不完整，请确保完整输出"）。
3. **部分结果挽救**：超时（30 分钟上限）前 2 分钟，worker 主动向 runtime 注入"请立即总结目前已完成的部分作为最终输出"；失败终态时若 timeline 中有可用的中间产物（已写入 workspace 的文件），卡片展示"虽然失败，但产生了这些中间产物"并可下载。
4. **失败卡片可行动**：终态失败卡片提供两个按钮——【原样重试】（同 context 重新入队）和【修改后重试】（指令预填进输入框）。

### 改成什么样
临时故障教师根本感知不到（自动重试成功）；真失败时也能拿到中间产物 + 一键重试，从"白等"变成"小挫折"。

### 验收标准
- 模拟 runtime 503：任务自动重试并最终成功，事件流含重试记录，教师无需操作。
- 模拟超时：最终输出包含"部分完成总结"而非空白失败。
- 失败卡片两个重试按钮均工作，重试任务继承原 context。
- 重试不会无限循环（最多 1 次自动 + 人工触发不限但每次都重新排队）。
- 上线一个月后统计：教师可见的"硬失败"（无任何产出的 failed）占比较基线下降 ≥50%。

### 测试形式与预估结果
- **单测**：错误分类器（给定错误样本断言分类）、重试计数、context 继承，约 10 个用例。预估全绿；错误分类样本库需要从生产日志攒，首版覆盖率可能只有常见错误的 70%。
- **集成**：stub runtime 按脚本先 503 后成功，断言全链路。
- **故障演练**：staging 人为掐断模型 API 60 秒，观察自愈表现。

### 偏差处理
- 若"超时前注入总结指令"在 deepseek-tui 上不可行（不支持运行中注入）→ 改为缩短任务软上限（25 分钟）+ prompt 预先约定"接近时限请先输出已完成部分"——把自救要求前置到任务开始。
- 若自动重试放大模型 API 费用 → 重试仅限 transient 类且全局每小时重试预算上限（如 10 次），超出则直接失败但文案说明"系统繁忙，请稍后用重试按钮"。
- 若错误分类经常误判 → 宁可保守：分类不确定时按 fatal 处理（不自动重试，但保留人工重试按钮），避免错误指令被反复执行。

## 全局验收与发布节奏

### 阶段门禁
每个阶段（P1→P2→P3）发布前必须通过：
1. `DB_ENGINE=sqlite python -m unittest`（全量单测）+ 新增用例全绿。
2. P03 Playwright harness 全链路截图评审（无布局回归、无 console error）。
3. `npm run build` + `npx tsc --noEmit` 干净。
4. 本地 compose（生产镜像）跑通双任务并发 30 分钟，内存/CPU 在预算内。
5. 部署用 `deployment/deploy_remote.ps1` 标准流程，部署后 curl 验证新接口与静态资源版本。

### 度量基线（上线前先采集，用于验证改进效果）
- 任务平均完成时长、排队时长分布
- 任务失败率 / 取消率（取消常意味着"等不及"）
- Agent 工具调用 SQL 首次成功率
- 每周 Agent 任务总量与人均任务数（易用性提升的最终指标是**用得更多**）

### 全局偏差原则
任何条目实施中发现复杂度远超预估时，按以下顺序裁剪——**先砍实现的优雅，再砍能力的完整，最后才砍易用性**：
1. 实时推送可降级为短轮询（体验近似）；
2. 自动化可降级为半自动（一键确认替代全自动）；
3. 智能可降级为预置模板（少而稳的选项替代自由编排）；
4. 绝不接受的降级：黑盒等待、裸错误信息、没有人工兜底出路。
