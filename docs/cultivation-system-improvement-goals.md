# 修为体系改进目标文档

> 版本：v1.0（2026-06-12）
> 范围：修为（学习进度）体系全链路 —— 学生端每课堂修为进化 / 作业提交 / 互动 / 行为分析，教师端统计与预警，以及支撑这一切的 AI 调度与使用方式。
> 核心代码：`classroom_app/services/learning_progress_service.py`（约 2800 行，修为计算 + 破境试炼）、`behavior_tracking_service.py`、`student_insight_service.py`、`personalized_learning_path_service.py`、`dashboard_service.py`；前端 `static/js/learning_progress.js`、`cultivation_identity.js`、`templates/learning_path.html`、`templates/classroom_main_v4.html` 修为区块。
> 总体方向：**易用性、教学创新性、美观性**，同时优先补齐架构地基（快照化、事件化、AI 异步化），不给未来埋坑。

---

## 现状基线

修为体系当前的能力与机制：

1. **修为分（0–100）**：材料研读 45 + 作业考试 35 + 互动求助 15 + 稳定投入 5，四项加权（`_build_learning_metrics`）。材料完成判定靠滚动比例 ≥0.86 / 活跃 ≥180s / 累计 ≥300s 的启发式。
2. **十层境界**：未入道 → 启蒙入门 → … → 可为人师（`LEARNING_LEVELS`），各层有解锁分、主题色、证书名、称号；境界必须逐层通过"破境试炼"晋升。
3. **破境试炼**：达到解锁分后，AI 按学生画像 + 境界范围内课程知识生成个性化试卷（`create_personal_stage_exam`），AI 批改 ≥80 分发证书（道印/灵契）。
4. **班级定位**：修为山脉图（`build_student_class_position`）+ 排名播报（登顶/前茅/进步空间/尚需努力）。
5. **教师端**：班级修为总览（`build_class_learning_overview`：汇总卡、境界分布、点将台）+ 学生洞察页（六维雷达、支持画像、作品集）+ 布尔型 `needs_attention` 标记。
6. **学习路径页**（`/learning-path`）：十层修为地图 + 每课程发力点 + 路径动作卡（复盘/下次行动）。

已具备的优点：境界叙事完整统一、破境试炼个性化、教师端数据维度齐全。下文改进均为增量，不推翻"修仙宗门"世界观与现有信息架构。

**已知架构债（本文档第 0 章优先处理）**：

- `refresh_student_learning_state` 在**每次读取时**全量重算四项指标（约 7 条聚合 SQL）并向 `learning_stage_status` 写 10 行 upsert——读路径有写放大；
- `build_student_class_position` 为画山脉图对**全班每个同学**调一遍 `_build_learning_metrics`——学生每刷一次课堂页是 O(全班 × 7 条 SQL)，200 人并发下是 2c/4GB 服务器的主要风险点；
- `create_personal_stage_exam` 在 **HTTP 请求路径内同步等待 AI 出卷**（`learning.py:247`，timeout 300s）——学生点"破境"后浏览器干等，且全班集中破境时 AI 服务被瞬时打满；
- 修为分是黑盒函数输出，**没有变动流水**——学生不知道分数为什么涨/不涨，教师无法审计，前端也没有"修为增长"动效的数据源。

## 0. 架构地基（先行，其余改进的依赖）

### G1. 修为快照表：读写分离，读路径零重算

- **为什么改**：如上所述，修为读取 = 全量重算 + 写库，班级山脉图更是全班重算。这既是性能炸弹（200 并发 / 2c4GB），也堵死了后续所有创新——成长曲线、周报、趋势统计都需要"某时刻的修为值"可被廉价读取，而现在每读一次都是一次昂贵的即时计算，且历史值根本不存在。
- **怎么改**：
  1. 新表 `learning_progress_snapshots`（offering_id, student_id, score, components_json, metrics_json, level_key, calculated_at，唯一键 offering+student），由现有 `refresh_student_learning_state` 的计算结果落库——计算逻辑不动，只把"算"和"读"分开。
  2. 新增 `get_student_learning_state(conn, offering_id, student_id)`：优先读快照；快照缺失或带脏标记时才现场重算并回写。所有**读路径**（dashboard、classroom 页、山脉图、教师总览、学生洞察、AI 提示词注入）一律改调它。
  3. 脏标记机制：在写入行为发生处（材料进度上报、提交作业、批改完成、聊天/行为事件批量落库）把 `(offering_id, student_id)` 标脏（快照表加 `dirty` 列即可，不引入新组件）。标脏后由**统一调度器**注册一个低频任务（每 2–5 分钟）批量重算脏行；学生本人触发的关键动作（交作业、材料完成）在请求内同步重算自己一行，保证"做完任务立刻看到修为涨"的即时反馈不丢。
  4. `build_student_class_position` 与 `build_class_learning_overview` 全部改读快照，单条 SQL 取全班分数。
- **改成什么样**：学生页/教师页的修为读取从"O(全班) 重算"降为一次索引查询；修为值至多延迟几分钟（本人动作即时），换来全链路可承载 200 并发。
- **验收标准**：
  - [ ] 课堂页与教师总览的请求路径内不再出现 `_build_learning_metrics` 调用（仅调度器与本人动作触发）；
  - [ ] 山脉图、点将台数据与重算结果一致（同一学生两条路径分数差 = 0）；
  - [ ] seed 60 人班级时，课堂页修为区块的 SQL 查询次数 ≤ 5（现状约 60×7）；
  - [ ] 快照缺失时自动回填，不出现 500 或空白。
- **测试形式与预估结果**：服务层单测（标脏→调度重算→读快照一致性；并发标脏不丢失）；用 P03 harness seed 大班级对比改造前后查询计数（SQLite 可用 `set_trace_callback` 计数）。预估 2–3 轮：脏标记的触发点容易漏（行为事件批量落库处最易漏），需要靠一致性测试逐个补。
- **偏差处理**：若某些触发点难以插桩（如第三方批改回调），兜底用调度器"全班低频轮询重算"（每 15 分钟）保证最终一致——宁可延迟也不回退到读时重算。若发现个别页面需要严格实时（如破境资格判定），仅该判定点保留现场重算，范围限定单个学生。

### G2. 修为流水账（cultivation ledger）：每一分都有出处

- **为什么改**：修为分对学生是黑盒（"为什么我学了一晚上才涨 0.3？"），对教师不可审计（无法回答家长/学生质疑），对前端是死数（没有"+2.4 修为"的脉动反馈可做），对防刷也无从下手（无法回查异常增长）。透明度是游戏化系统可信度的根，现在缺失。
- **怎么改**：
  1. 新表 `cultivation_score_events`（offering_id, student_id, event_type, delta, component, source_ref, created_at）。在 G1 的重算流程里，比较新旧 components 差值 ≥0.1 分即记一条流水，event_type 标注归因（material_progress / submission_graded / interaction / consistency / recalibration）。
  2. 学生端 API `GET /api/learning/score-events`：返回近 30 天流水（分页），课堂页修为面板加"修为日志"折叠区，逐条显示"6-10 完成《第 3 章》研读 +1.8"。
  3. 流水只追加不修改（审计语义）；调度器任务按月归档 90 天前流水到汇总行，控制表体积（70GB 磁盘约束）。
- **改成什么样**：学生点开修为数字能看到一本"修行账本"；前端动效、周报、防刷审计从此有统一数据源。
- **验收标准**：
  - [ ] 完成一份材料后流水出现对应 material 条目，delta 与快照分差一致；
  - [ ] 流水累加 ≈ 当前修为分（允许 ±0.5 取整误差）；
  - [ ] 90 天归档任务在 seed 数据上正确合并且明细被清理。
- **测试形式与预估结果**：单测覆盖 delta 拆分与归档；P03 截图断言日志区渲染。预估 2 轮：components 间的舍入归因会有小数对不齐问题，第二轮统一用"先算总 delta 再按分量比例拆"解决。
- **偏差处理**：若流水量在真实班级超预期（互动分抖动频繁产生碎条目），把记账阈值从 0.1 提到 0.5 并合并同日同类条目；若学生对"分数下降"条目（recalibration）产生困惑，UI 隐藏负向条目仅保留教师端可见，业务表照记。

## A. 学生端易用性

### A1. "差几分、最快怎么补"——可解释的破境路径面板

- **为什么改**：现在学生看到"修为 41.2 / 解锁 46"只知道差 4.8 分，不知道这 4.8 分从哪来最快。四项权重虽然展示了（`rules.score_weights`），但学生要自己心算"完成一份材料值多少分"，易用性断在最后一公里——知道差距却无法行动。
- **怎么改**：
  1. 服务端基于现有 metrics 推导"增量机会清单"：未完成材料（每份 ≈ 45/材料总数 分）、未提交作业（按 task 权重折算）、互动缺口（距离上限还差几次提问）。新增纯函数 `build_score_opportunities(metrics, next_stage)`，输出 ≤4 条按性价比排序的行动项，每条带预估增量与落地链接（材料/作业直达）。
  2. 课堂页修为卡与 `/learning-path` 焦点卡接入：进度条下方显示"距 {下一境界} 还差 4.8 分 · 最快路径：完成《第 5 章》研读（约 +3.1）"。
  3. 预估值口径标"约"，避免承诺精确值（互动/稳定分有抖动）。
- **改成什么样**：每个未破境学生在任何修为入口处都能看到一条"具体、可点击、带预估收益"的下一步，把进度条从展示物变成导航器。
- **验收标准**：
  - [ ] seed 学生（材料缺 2、作业缺 1）能看到正确排序的机会清单且链接可达；
  - [ ] 全部完成仍未达分时（依赖互动分场景）给出互动类建议而非空白；
  - [ ] 已可破境（challenge_ready）的学生不显示该面板，显示破境入口。
- **测试形式与预估结果**：`build_score_opportunities` 纯函数参数化单测（边界：零材料课程、全完成、只差互动分）；P03 截图。预估服务端 1 轮通过（纯函数易测），文案与排版 1 轮微调。
- **偏差处理**：若预估增量与实际涨分偏差大引发学生不信任，第一时间收窄文案为定性表述（"完成材料是当前最快的提升路径"）去掉数字，函数保留，等 G2 流水数据校准系数后再恢复定量。

### A2. 破境试炼异步化：点了不用等，好了喊你来

- **为什么改**：`POST /api/learning/.../stage-exam` 在请求内同步等 AI 出卷（最长 300s），学生面对转圈不敢刷新；超时/失败后只把状态打回 `challenge_ready`，学生必须自己发现并手动重试。这是修为体系里体验最差、也最危险（瞬时打满 AI 服务）的一环。平台已有统一调度器、agent 任务自动重试、消息中心完成通知三套基建，本应复用。
- **怎么改**：
  1. 路由改为"登记即返回"：写入 `generating` attempt 后立刻返回 `{status: "generating"}`，实际出卷由统一调度器的即时任务执行（复用 `scheduled_task_service` 的 handler 注册模式，新 handler `stage_exam_generation`）。
  2. 失败自动重试 ≤2 次（间隔退避），仍失败则置 `failed` 并发消息中心通知（"试炼生成失败，已为你保留破境资格，可重新发起"）；成功时发"破境试炼已就绪"通知，链接直达 `/exam/take/{assignment_id}`，复用既有 agent 完成通知的跳转模式。
  3. 前端修为卡的"生成中"态加轮询（复用现有 attempt status 字段）+ 预计等待文案（"约 1–3 分钟，可先去做别的，好了会在消息中心提醒你"）。
  4. handler 内做并发闸：同一 offering 同时出卷数 ≤2（队列排队），保护 2c4GB 与 AI 服务。
- **改成什么样**：点"破境"立即得到反馈并可以离开页面；成败都有通知闭环；全班同时破境时系统平稳排队而非雪崩。
- **验收标准**：
  - [ ] 发起破境的 HTTP 响应 < 1s；
  - [ ] mock AI 延迟 60s 场景下学生可正常浏览其他页，完成后收到消息中心通知；
  - [ ] mock AI 连续失败时重试 2 次后置 failed + 失败通知，资格保留可重新发起；
  - [ ] 同 offering 并发 10 个出卷请求时 AI 侧并发 ≤2。
- **测试形式与预估结果**：服务层单测（mock ai_client：成功/超时/格式错三分支 + 重试计数）；调度 handler 集成测试复用 scheduler 既有测试模式。预估 2–3 轮：状态机迁移（generating→failed→challenge_ready 的恢复路径）和并发闸是易错点。
- **偏差处理**：若调度器 worker 的任务延迟在生产高峰不可接受（出卷迟迟不开始），允许 handler 旁路：请求路径直接 `asyncio.create_task` 后台执行 + 任务表登记兜底，调度器只负责扫描卡死任务（generating 超 10 分钟）做恢复。状态表是真相源，执行载体可替换——这正是不埋坑的关键。

### A3. 材料研读完成判定升级：从"挂机可过"到"轻量检验"

- **为什么改**：当前材料完成 = 滚动 86% 或活跃 180 秒，纯行为启发式，挂机+拖滚动条即可刷满 45 权重的材料分。这伤害两类学生：认真学的人觉得不公平，投机的人骗过系统却没学到东西；教师端"材料完成 92%"的统计也随之失真。
- **怎么改**：
  1. 行为判定保留为"已研读"，新增第二档"已掌握"：材料解析时（已有 `ai_parse_result_json` 管线）由 AI 预生成 2–3 道单选微检验（存材料表新列 `check_questions_json`，**一次生成全班复用**，无每人成本）；学生读完弹出可跳过的"心法检验"（答对 ≥2 升为已掌握）。
  2. 修为材料分拆为：已研读得该材料分值的 70%，已掌握得 100%。旧数据全部按"已研读"折算 70%？——不行，这会让全班存量分数集体下降。**迁移策略：存量已完成材料一次性视同"已掌握"**，新规则只对文档发布后的新完成生效，并在修为日志（G2）注明规则版本。
  3. 检验可无限重试、不计入成绩、错题即时给解析——定位是巩固工具而非考试，文案明确"检验不扣分"。
  4. AI 生成失败的材料自动降级为单档判定（行为完成即满分），不阻塞材料发布。
- **改成什么样**：刷时长最多拿七成材料分；认真读完顺手做 3 道题的学生拿满分并得到即时巩固反馈；教师统计多一层"掌握率"维度。
- **验收标准**：
  - [ ] 新材料上传后检验题在后台生成完成（调度器任务），生成失败不影响材料可见；
  - [ ] 跳过检验得 70%、通过得 100%，修为日志条目正确标注；
  - [ ] 存量数据迁移后无任何学生修为分下降；
  - [ ] 检验弹层支持键盘操作与 reduced-motion。
- **测试形式与预估结果**：服务层单测（两档折算、存量迁移、AI 失败降级）；P03 走通"读材料→弹检验→答题→修为日志出现条目"全链路。预估 3 轮：检验题生成质量需要 1–2 轮提示词打磨（选项合理性、答案唯一性），用既有试卷 normalize 思路做结构校验。
- **偏差处理**：若 AI 生成的检验题质量不稳（歧义、超纲），给教师材料管理页加"查看/禁用检验题"开关，被禁用的材料回退单档判定；若学生反感弹层打断，把触发改为材料页尾部内联卡片（不弹层），核心是两档分值结构而非交互形式。

### A4. 互动分防刷与正向引导：数质量，不数条数

- **为什么改**：互动分按消息条数/@次数计数（LIKE '%@助教%'），学生发 12 条"哈哈"就能拿满聊天分量。上限封顶（min(n, cap)）控制了刷分上界，但无法区分灌水与有效提问；更糟的是它教学生"互动 = 凑条数"，与教学目标背道而驰。
- **怎么改**：
  1. 复用既有行为分析管线（`classroom_behavior_*`、AI 行为画像已在跑）：在每日/每次行为画像生成时，让 AI 顺带输出 `interaction_quality`（0–1，评估提问具体性、与课程相关性），存入行为画像表新列——**不新增 AI 调用，搭画像任务的便车**。
  2. 互动分公式改为 `条数分 × (0.5 + 0.5 × quality)`：无画像时 quality 默认 0.7（轻微保守），有画像按质量浮动。灌水学生最多拿一半互动分。
  3. 学生端"修为说明"弹层补一句透明规则："互动看质量：具体的问题、对讨论的贡献比条数更重要"。不公开具体系数，避免反向工程。
- **改成什么样**：互动分从"计数器"变成"质量加权计数器"，引导学生提真问题；改动幅度被 0.5 下限兜底，不会出现互动分腰斩的剧烈波动。
- **验收标准**：
  - [ ] 无画像学生互动分变化幅度 ≤15%（0.85 系数兜底验证）；
  - [ ] mock 高/低质量画像时互动分正确浮动；
  - [ ] 互动 15 权重的总盘不变，其他分量零影响。
- **测试形式与预估结果**：`_load_interaction_metrics` 单测注入 quality 参数化断言；行为画像 prompt 扩展用既有画像测试夹具回归。预估 2 轮：quality 字段在画像 JSON 中的稳定输出需要一轮提示词约束（schema 校验兜底）。
- **偏差处理**：若 AI quality 评分自身不稳定（同样本波动大），退一步用确定性代理指标（消息平均长度、含课程关键词比例）算 quality，公式结构不变；若学生察觉并产生"被 AI 打分"的不适，教师端开关可按 offering 关闭质量加权，回退纯计数。

## B. 教学创新性

### B1. 修为成长曲线与"修行周报"

- **为什么改**：修为只有"当前值"，没有时间维度——学生看不到自己一个月来的成长轨迹（最强的坚持动机），教师看不到班级趋势（只能看静态分布）。G1 落地后历史数据近乎免费，这是性价比最高的创新点。
- **怎么改**：
  1. 统一调度器注册每周日晚任务：把全班快照（G1）按周存入 `cultivation_weekly_snapshots`（offering, student, week_start, score, components_json, level_key）。
  2. 学生课堂页修为卡新增"成长曲线"迷你折线（近 8 周，纯 SVG 内联渲染，复用洞察页雷达的服务端 SVG 坐标计算模式，不引图表库）。
  3. 修行周报：周一清晨调度任务为活跃学生生成一条消息中心通知，模板化拼装（本周修为 +X、完成材料 N 份、境界变化、下周一句建议来自 A1 机会清单）——**第一版纯规则文案不调 AI**，数据全部来自周快照差值与 G2 流水，保证零幻觉、零成本。
  4. AI 美化作为二期开关：周报文案可选走 AI 润色（批量、低优先级队列），规则版始终是降级兜底。
- **改成什么样**：学生每周一收到一份有具体数字的"修行总结"，课堂页能看到自己的成长折线；坚持被看见，反馈周期从"实时分数"补上"周节奏"一层。
- **验收标准**：
  - [ ] 周快照任务幂等（重跑同一周不产生重复行）；
  - [ ] 折线在 1 周数据（单点）、8+ 周数据下渲染正常；
  - [ ] 周报数字与快照差值一致；零活动学生不发周报（避免打扰）；
  - [ ] 首页请求路径零新增计算（全部读周表）。
- **测试形式与预估结果**：调度 handler 单测（幂等、周界计算跨月/跨年）；周报文案模板参数化单测；P03 折线截图。预估 2 轮：周界与时区/学期边界的 off-by-one 通常要一轮修。
- **偏差处理**：若周报被学生视为打扰（消息中心噪音），改为不推送、仅在 dashboard 渲染"本周修行"卡（拉取制）；若折线在修为长期不动的学生那里显得尴尬（直线），低于 2 个数据变化点时隐藏折线只显示文字摘要。

### B2. 破境失败后的"闭关清单"：把失败变成路径

- **为什么改**：试炼 ≥80 通过发证书，<80 失败后系统只是把状态打回，学生得到一个分数和挫败感，却不知道"差在哪、补什么"。AI 批改其实产出了逐题评语——这些最有教学价值的数据在失败场景下没有被组织成行动。
- **怎么改**：
  1. `handle_stage_exam_grading_complete` 的失败分支扩展：把 AI 批改的逐题结果（已有）按错题聚合，匹配境界范围内的课堂章节/材料（`_build_course_knowledge_snapshot` 已有章节-材料映射），生成 3–5 条"闭关清单"写入 `personalized_learning_path_service` 的既有路径动作结构（learning_path 页面与置顶/完成/复盘交互**全部现成**，零新 UI）。
  2. 每条清单项：错题知识点描述 + 直达材料/章节链接 + "弄懂后写一句复盘"（复用 path-card 的 reflection 字段）。
  3. 失败通知文案重写：从"未通过"改为"试炼显示 X 处薄弱点，闭关清单已生成"，链接到 learning-path 过滤视图。
  4. 二次破境的出卷提示词注入上次错题摘要（已有 metadata_json 可挂载），让 AI 对薄弱点变式重考而非简单换题。
- **改成什么样**：破境失败从死胡同变成"系统帮你把差距拆成了 4 件事"；二次试炼针对性重考，通过即代表薄弱点真正补上——考试闭环升级为学习闭环。
- **验收标准**：
  - [ ] mock 失败批改结果后 learning-path 出现对应清单项且链接正确；
  - [ ] 清单项匹配不到材料时降级为纯文字建议（不出现死链）；
  - [ ] 二次出卷 prompt 包含上次错题摘要（单测断言 prompt 内容）；
  - [ ] 通过场景零变化（不生成清单）。
- **测试形式与预估结果**：服务层单测（错题聚合、章节匹配、降级）；P03 走通失败→清单→复盘保存链路。预估 3 轮：错题→章节的匹配准确率需要调（批改评语里知识点表述自由），先用材料名/章节标题关键词匹配，命中率低再考虑挂 AI 归类。
- **偏差处理**：若知识点匹配噪音大（清单指向错误材料），把链接降级为"该境界全部材料列表"入口，清单只保留 AI 评语原文的薄弱点描述——准确的弱点描述 + 宽泛的入口，好于错误的精确链接。

### B3. "传功"机制：让高境界学生的帮助行为有修为回报

- **为什么改**：最高境界叫"可为人师"，但体系里没有任何"为人师"的行为通道——互动分只认提问，不认回答。班级里高境界学生答疑帮同学是真实存在的高价值行为，当前体系不可见、无激励；同时这是把游戏化从"个人刷分"拉向"共同体学习"的关键一步，教学创新价值最高。
- **怎么改**：
  1. 课堂聊天消息增加轻量"有用"标记（仅提问者本人与教师可标，复用聊天消息的既有操作菜单模式）；被标记的回答者记一条 `peer_help` 行为事件（既有 `classroom_behavior_events` 表，新 action_type，零新表）。
  2. 互动分公式加入 peer_help 分量（与提问同栏，min(n,3)/3 × 一定比重，从现有互动 15 分内部重新切分，**不扩总盘**）；tier ≥6（基本掌握）后 peer_help 权重翻倍——境界越高，助人越值钱，呼应"可为人师"。
  3. 教师端点将台学生行加"助人 N 次"标记；月度"传功榜"进班级总览（≤3 人，只展示正向榜，不设末位榜）。
  4. 防互刷：同一对学生每日互标计 1 次有效；教师标记不限。
- **改成什么样**：会的学生教不会的学生在体系内被看见、被奖励；"可为人师"从一个称号变成一条真实的行为路径。
- **验收标准**：
  - [ ] 提问者标记后回答者互动分上涨且修为日志可见"助人"条目；
  - [ ] 同对学生当日重复标记不重复计分；
  - [ ] 非提问者/非教师看不到标记按钮；
  - [ ] tier ≥6 学生的 peer_help 增量约为低 tier 的 2 倍。
- **测试形式与预估结果**：行为事件与计分单测（含防刷对）；P03 双账号交互测试（提问→回答→标记→分数变化）。预估 3 轮：聊天 UI 的标记交互（islands/旧 JS 双栈）是工作量大头，先在旧 `classroom_interactions.js` 栈实现。
- **偏差处理**：若出现组团互刷（A 问 B 答循环），收紧为"仅教师标记计分"运行一学期观察；若标记功能使聊天 UI 复杂化（违背 declutter 约定），改为教师在消息长按菜单里单入口操作，学生侧完全不见操作只见结果。

### B4. 教师可调的课程修为权重（带版本记录）

- **为什么改**：45/35/15/5 权重硬编码，对所有课程一刀切。实验课材料少作业重，研讨课互动是核心——教师对"什么算学得好"的定义被代码绑死，这限制了体系作为教学工具的适配性，也是教师最可能提出的第一个需求。
- **怎么改**：
  1. `class_offerings` 配置（或既有 offering 设置 JSON 列）增加 `cultivation_weights`，缺省即现行 45/35/15/5；教师课堂管理页新增"修为权重"设置卡，四滑块联动强制总和 100，附三个预设（讲授型/实验型/研讨型）。
  2. `_build_learning_metrics` 读 offering 权重；**快照与流水（G1/G2）记录所用权重版本**，权重修改后全班标脏重算并在修为日志记一条 recalibration 条目——这是不埋坑的关键：任何历史分数都能解释"当时按什么规则算的"。
  3. 学生端 `rules.score_weights` 已是动态下发，前端零改动自动跟随。
  4. 权重修改限频（每 offering 每 7 天 1 次）+ 修改时弹层预览"全班分数将如何变化"（用快照数据即时模拟，G1 后廉价）。
- **改成什么样**：教师能让修为体系长成自己课程的形状；学生看到的规则说明自动同步；任何时刻的分数都可追溯到规则版本。
- **验收标准**：
  - [ ] 未配置课程行为与现状完全一致（默认值回归零差异）；
  - [ ] 修改权重后全班重算、学生端权重说明同步、日志出现 recalibration；
  - [ ] 总和 ≠100 的提交被拒（前后端双校验）；
  - [ ] 7 天限频生效。
- **测试形式与预估结果**：服务层单测（默认回退、重算触发、版本记录）；P03 教师改权重→学生页验证链路。预估 2 轮：预览模拟的口径与实际重算一致性需要一轮对账。
- **偏差处理**：若教师滥调权重引发学生公平性质疑（学期中大改），把限频收紧为"开课 2 周后锁定，改动需在下一学期生效"；若滑块交互在移动端难用，退化为三预设单选 + 自定义数字输入。

## C. 美观性

### C1. 破境时刻的仪式感：证书与晋升动效统一升级

- **为什么改**：境界体系铺垫了完整的命名/称号/主题色，但"破境成功"——整个体系情绪价值最高的时刻——目前只是状态翻转 + 一条通知，没有与十层叙事匹配的视觉高光。仪式感是游戏化留存的核心机制，这里是美观性投入回报最高的位置。
- **怎么改**：
  1. 破境通过后首次回到课堂页/dashboard 时播放一次性"晋升揭幕"层（境界主题色光晕 + 证书卡浮现 + 称号变更动画，纯 CSS animation + 少量自包含 JS，`prefers-reduced-motion` 下退化为静态卡淡入）；播放标记记 localStorage + 服务端 certificate 的 `revealed_at`，确保只播一次、换设备不重播。
  2. 证书卡重绘：以 `LEARNING_LEVELS` 的 theme 为基色生成分层 SVG 底纹（每 tier 一套渐变 + 纹样强度递增，CSS 变量着色适配明暗主题），含证书名、称号、课程宗门名、日期；提供"保存图片"按钮（canvas 导出，社交分享素材）。
  3. 山脉图 polish：当前学生点高亮脉冲、登顶者旗标、自己位置的称号气泡——只动 CSS 与既有 SVG 输出，不改数据结构。
  4. 全部样式进 `ui-system.src.css` 底部 polish 层，遵守既有约定。
- **改成什么样**：破境是一个值得截图发朋友圈的时刻；证书从一行记录变成一件可收藏、可分享的"道具"；视觉语言与十层世界观严格一致。
- **验收标准**：
  - [ ] 晋升揭幕只播一次（刷新/换页不重播），reduced-motion 下无动画但信息完整；
  - [ ] 10 个 tier 的证书卡在明暗主题下均无对比度问题（截图全套）;
  - [ ] 导出图片在移动端可用（长按保存兜底）；
  - [ ] 动画期间页面可交互（不阻塞，可点跳过）。
- **测试形式与预估结果**：P03 截图矩阵（10 tier × 明暗 × reduced-motion）；revealed_at 标记单测。预估 3 轮：动效节奏与"克制不浮夸"的平衡需要视觉迭代，第一轮通常过重。
- **偏差处理**：若 canvas 导出在微信内置浏览器兼容性差，降级为"长按图片保存"的纯 `<img>`（服务端预渲不做，直接 SVG data-url）；若动画被反馈幼稚/过度，保留证书卡浮现、砍掉光晕粒子——卡片本身是底线交付物。

### C2. 修为信息层级重排：一眼境界、两眼差距、三眼路径

- **为什么改**：课堂页修为区块当前把分数、进度条、境界名、排名播报、试炼入口平铺，信息密度高但层级平——学生扫一眼抓不住重点；dashboard、classroom、learning-path、profile 四处的修为展示样式各异（各自模板各写一套），视觉不统一且维护成本翻倍。
- **怎么改**：
  1. 抽一个统一的修为卡 partial（`partials/cultivation_card.html` + 配套 CSS 组件类），三档尺寸：徽章（行内 chip，点将台/聊天头像用）、紧凑卡（dashboard 课程脉搏用）、完整卡（课堂页/learning-path 用），数据契约统一为 `public_level_payload` + 快照字段。
  2. 完整卡信息层级固化为三层：**第一层**境界徽记 + 称号（最大视觉权重，theme 色）；**第二层**进度环 + "距下一境差 X 分"（A1 的机会入口挂这里）；**第三层**折叠的修为日志/规则说明。排名播报降为第二层的一行小字（避免排名焦虑占据主视觉）。
  3. 四处入口逐个迁移到 partial，迁移一处回归一处截图。
- **改成什么样**：全站修为展示同一张脸、三档尺寸；学生视线动线 = 我是谁 → 差多少 → 做什么，与 A1 形成完整漏斗。
- **验收标准**：
  - [ ] 四个页面修为区块全部来自同一 partial（grep 无残留旧标记结构）；
  - [ ] 三档尺寸在 390px/768px/1280px 截图无溢出错位；
  - [ ] 迁移前后数据展示零丢失（字段对照清单核对）；
  - [ ] 排名播报不再出现在第一视觉层级。
- **测试形式与预估结果**：P03 截图矩阵（4 页 × 3 视口）；partial 渲染单测（三档 × 有无证书/可破境态）。预估 3–4 轮：四处迁移各有一轮微调，learning-path 的焦点卡耦合最深放最后。
- **偏差处理**：若某入口的特殊需求难以塞进三档（如 profile 页需要跨课程汇总视图），允许该页在 partial 外包一层页面级容器，但境界徽记/进度环组件必须复用——组件统一是底线，布局可以例外。

## D. 教师端统计与预警

### D1. 班级修为趋势看板：从"现在怎样"到"正往哪走"

- **为什么改**：`build_class_learning_overview` 给的全是时点快照（平均分、分布、活跃数），教师看不到方向——平均修为 52 是在爬升还是停滞？哪一周全班集体掉速（通常对应教学事件）？没有趋势，统计就只能用于点名，不能用于教学决策。
- **怎么改**：
  1. 基于 B1 周快照表，班级总览新增"趋势"区：班级平均修为 8 周折线、境界分布周对比（堆叠条 ×2：上周 vs 本周）、四分量均值环比箭头（材料/任务/互动/稳定各自 ↑↓ 与幅度）。全部服务端聚合 SQL + 内联 SVG，零图表库。
  2. 汇总卡补环比小字（"平均修为 52.3，较上周 +1.8"）。
  3. "掉速学生"列表：本周修为增量为 0 且上周 >0 的学生（停滞侦测），作为 D2 预警的一个信号源先在这里可视化。
- **改成什么样**：教师打开总览 10 秒内回答三个问题：班级整体在进步吗？哪个分量拖后腿？谁这周停下来了？
- **验收标准**：
  - [ ] 趋势区数据与周快照表逐周核对一致；
  - [ ] 不足 2 周数据时显示"积累中"占位而非空图；
  - [ ] 总览页响应时间不因趋势区上升 >100ms（全走周表索引查询）；
  - [ ] 掉速列表不含本周本来就无任务安排的误报（无新任务周全班停滞时给班级级提示而非逐人标记）。
- **测试形式与预估结果**：聚合 SQL 单测（跨周边界、转学/退课学生处理）；P03 截图。预估 2 轮：掉速误报的口径需要一轮真数据校准。
- **偏差处理**：若"无任务周全班停滞"的误报判定复杂化，第一版掉速列表只做"连续 2 周零增量"的强信号，弱信号留给 D2 的综合预警去权衡。

### D2. 分级预警 + 处理闭环：从布尔标记到工作流

- **为什么改**：现在的预警是单一布尔 `needs_attention`（有待办/完成率<50%/零活跃 三选一即标），不分轻重缓急，教师面对"待关注 18 人"无从下手；预警也没有去向——看见了、然后呢？没有处理动作和消音机制，预警很快沦为常驻噪音被无视。
- **怎么改**：
  1. 预警引擎独立成 `cultivation_alert_service`：规则注册表模式（每条规则 = 取数函数 + 级别 + 文案模板），首批规则：**L1 提示**（单任务临期未交）、**L2 关注**（修为连续 2 周零增量 / 完成率 <50%）、**L3 干预**（连续 7 天零活动 / 破境试炼 2 次失败 / 修为分位从前 50% 跌出后 25%）。由统一调度器每日清晨跑一次，结果写 `cultivation_alerts` 表（学生、规则、级别、状态、证据 JSON）。
  2. 教师端总览新增预警收件箱：按级别分组，每条带证据摘要（"连续 9 天无任何课堂活动，最后活跃 6-2"）与三个快捷动作——**私信学生**（预填关切模板）、**记备注**（写入既有共享备注）、**标记已处理/本周静音**。处理状态落库，已处理项不再重复弹出（同规则同学生 7 天冷却）。
  3. L3 级新增可选的消息中心推送给教师（offering 级开关，默认开），复用既有通知模式。
  4. 点将台的 `status_tags` 改为读预警表（保持向后兼容的标签文案）。
- **改成什么样**：教师每天看到一个分好级、带证据、可一键行动、处理后会消失的预警收件箱；"预警"从形容词变成有生命周期的工单。
- **验收标准**：
  - [ ] 三级规则在 seed 场景下各自正确触发且互不重复（同一学生命中多规则时按最高级聚合展示）；
  - [ ] 标记处理后 7 天内同规则不再生成；
  - [ ] 快捷私信/备注动作落到既有表且学生侧体验无异常（私信正常收到）；
  - [ ] 预警生成是调度任务，教师页请求路径只读表。
- **测试形式与预估结果**：规则函数逐条参数化单测（边界日期）；调度 handler 幂等测试；P03 教师端收件箱交互链路。预估 3 轮：规则阈值需要一轮真实数据回放校准（用生产匿名导出在本地 seed 重放），冷却与聚合逻辑一轮。
- **偏差处理**：若某条规则误报率高（如假期导致全班 7 天零活动），规则注册表天然支持单条停用——先停规则保收件箱信誉，调阈值后再启用；若教师反馈收件箱仍是负担，默认折叠 L1 只展开 L2/L3，必要时 L1 直接停产。

### D3. 学生洞察页接通预警与流水：证据链一页打通

- **为什么改**：教师从预警点进学生洞察页后，看到的是与预警无关的通用面板（雷达、课程进度、作品集），需要自己把"为什么被预警"和页面数据对起来；G2 流水与 D2 证据已经存在，洞察页却没有呈现入口——数据有了，叙事断了。
- **怎么改**：
  1. 洞察页顶部加"当前预警"条（仅当该生有未处理预警时渲染）：级别、证据、与 D2 相同的快捷动作，处理后原地消失。
  2. 新增"修为时间线"区块：合并 G2 流水（取关键条目：境界变化、试炼通过/失败、大额增量）+ 预警事件 + 教师备注，按时间倒序的单列时间线（≤20 条），让教师 30 秒读完这个学生的"剧情"。
  3. 雷达图旁补四分量的班级均值虚线对照（数据来自总览均值，已有），让"互动 38" 有参照系。
- **改成什么样**：教师从预警进来，一页之内完成"看证据 → 看历史 → 采取行动"，不需要在三个页面之间拼图。
- **验收标准**：
  - [ ] 有/无预警两态渲染正确，处理动作与 D2 收件箱状态同步；
  - [ ] 时间线条目类型齐全且排序正确，超 20 条正确截断；
  - [ ] 雷达对照线在班级只有 1 人时隐藏（无统计意义）。
- **测试形式与预估结果**：时间线合并函数单测（多源排序、截断）；P03 截图。预估 2 轮，依赖 G2/D2 先行。
- **偏差处理**：若时间线信息量在活跃学生处爆炸（互动碎条目刷屏），收紧关键条目准入（境界/试炼/预警/备注四类，普通增量条目聚合为周摘要行）。

## E. AI 系统调度与合理使用

### E1. 修为体系 AI 调用全量入队：统一限流、削峰、可观测

- **为什么改**：修为链路的 AI 调用散落三种形态——破境出卷是请求内同步 httpx（A2 处理）、AI 批改走自己的提交管线、行为画像走另一套任务；没有统一的并发上限与优先级，全班破境 + 批改高峰叠加时 2c4GB 上的 AI 服务无差别过载，最先饿死的往往是学生正在等的那一个。平台已有统一调度器与 agent 任务表，缺的是修为侧全部接入 + 一个全局闸。
- **怎么改**：
  1. 在 ai_client 层（或薄包装）实现全局信号量 + 优先级队列：P0 交互等待类（学生正在等的批改/出卷）、P1 当日任务（预警扫描的画像更新）、P2 批量后台（材料检验题生成、周报润色）。总并发上限配置化（默认 3），P2 仅在队列空闲时执行。
  2. 修为链路所有 AI 调用点（出卷 A2、批改、画像、检验题 A3、周报 B1）逐一迁入，调用处只声明优先级与超时。
  3. 每次调用记 `ai_usage_log`（task_type、tokens 估算、耗时、成败、offering），为 E3 的用量看板与成本控制供数。
  4. 过载策略：队列深度超阈值时 P2 任务直接延期到夜间窗口（调度器改期而非丢弃）。
- **改成什么样**：AI 服务在任何高峰下按"学生在等的优先"的次序消化任务；每一次 AI 调用都有账可查；新增 AI 功能只需声明优先级即可获得限流保护——这是后续所有 AI 创新的安全网。
- **验收标准**：
  - [ ] 并发 20 个混合优先级任务时，P0 平均等待 < P1 < P2，且 AI 侧实际并发 ≤ 配置值；
  - [ ] P2 任务在队列高压下被改期且最终执行（不丢失）；
  - [ ] 全部修为链路调用点出现在 usage log 中（grep 直连 ai_client 的残留为零）；
  - [ ] 任一任务崩溃不影响队列继续消化。
- **测试形式与预估结果**：队列单测（优先级、信号量、异常隔离，mock 延迟注入）；集成测试跑混合负载断言次序。预估 3 轮：批改管线的迁移牵涉既有重试逻辑，是最复杂的一处。
- **偏差处理**：若批改管线迁移风险过高（牵涉已稳定的 agent 重试机制），允许批改暂留原管线、仅共享全局信号量（最小集成），队列化推迟到下一期——全局并发闸是必须项，统一队列是改进项。

### E2. 出卷质量自动护栏：重复检测 + 教师抽检入口

- **为什么改**：破境试卷由 AI 按学生个性化生成，但没有质量护栏：同一学生重考可能拿到近似卷（metadata 未注入历史题）、不同学生可能撞题（个性化失效）、超纲/答案错误只能靠学生事后发现。试炼决定证书发放，质量问题直接动摇整个境界体系的公信力。
- **怎么改**：
  1. 出卷后、发布前插入校验管线（纯本地零 AI 成本优先）：结构校验（已有 normalize）之上增加——与该生该境界历史卷的题干相似度检测（简单 n-gram/编辑距离，重复率 >40% 自动重新生成一次）、与同班同境界近 10 份卷的抽样查重（>60% 相似题占比时记质量事件，不阻塞）。
  2. 质量事件写表并在教师端课堂管理加"试炼卷抽检"入口：列出近期生成卷 + 质量标记，教师可预览试卷、一键作废重生成（作废自动通知学生）。
  3. 答案合法性强校验：radio answer 必须在 options 内、checkbox answer 是 options 子集——不合法直接重生成（最多 2 次，仍失败走 A2 的失败通知）。
  4. 出卷 prompt 注入"该生历史已考题干列表"（截断 1500 字），从源头降低重复。
- **改成什么样**：低质量卷在学生看到之前被拦截或标记；教师拥有最终质检权但默认无需介入；重考的学生确定拿到不一样的题。
- **验收标准**：
  - [ ] 注入重复 mock 卷时触发自动重生成且计数受限；
  - [ ] answer 不在 options 的 mock 卷被拦截；
  - [ ] 教师抽检页可预览/作废/重生成，作废后学生收到通知且资格保留；
  - [ ] 校验管线增加的出卷耗时 <2s（本地计算）。
- **测试形式与预估结果**：校验函数参数化单测（构造各类劣卷）；查重阈值用既有 seed 卷标定。预估 2–3 轮：相似度阈值需要一轮真卷标定避免误杀正常卷。
- **偏差处理**：若 n-gram 查重对中文题干误报高，先退到"完全相同题干"的精确去重（零误报底线），语义查重等 embedding 基建就绪再升级；若自动重生成造成出卷时长翻倍引发体验问题，重复检测降级为"标记不阻塞"，把拦截权交给教师抽检。

### E3. AI 用量透明看板与课程级预算

- **为什么改**：修为体系的 AI 消耗（出卷、批改、画像、检验题、周报）随学生活跃度线性增长，但当前没有任何用量视图——管理员无法回答"这学期 AI 花在哪了"，也没有手段在异常（如学生反复删卷重出）时止血。合理使用的前提是可见、可控。
- **怎么改**：
  1. 基于 E1 的 `ai_usage_log`，管理端新增"AI 用量"页：按 task_type / offering / 周的聚合表 + 趋势条，异常高亮（环比 ×3 以上标红）。
  2. 课程级软预算：offering 配置每周 AI 调用配额（默认宽松，如出卷 50 次/周），超额后 P2 类任务暂停、P0 类放行但记超额事件并通知管理员——**永远不在学生等待的路径上硬卡**，预算先用于观测与告警。
  3. 学生级防滥用：同一学生同一境界 24h 内出卷 ≤3 次（删卷重出计入），超出温和提示"今日试炼机会已用完，明日再来"——配合修仙叙事（"灵力需要恢复"）减少对抗感。
  4. 看板数据全部读 usage log 聚合，零额外打点。
- **改成什么样**：AI 成本从黑箱变成一张可下钻的账单；滥用被限频温和拦截；预算超限触发的是"人的判断"而非"学生的报错"。
- **验收标准**：
  - [ ] 看板数字与 usage log 逐条核对一致；
  - [ ] 学生第 4 次出卷被限频且文案友好、资格不受损；
  - [ ] 超预算时 P0 任务不受影响、管理员收到通知；
  - [ ] 看板页查询 <500ms（聚合走索引或预聚合周表）。
- **测试形式与预估结果**：限频与预算判定单测（跨日边界）；看板聚合 SQL 单测。预估 2 轮，依赖 E1 先行。
- **偏差处理**：若周配额默认值在真实课堂频繁误触（活跃班级正常用量超估），上调默认并把首学期定位为"纯观测不拦截"，用真实分布定阈值后再启用拦截；学生限频如引发投诉，给教师一个"为某生临时解限"的按钮。

## 实施顺序与依赖

```
第一阶段（地基）：G1 快照 → G2 流水 → A2 试炼异步化 → E1 AI 入队
第二阶段（体验）：A1 机会面板 → C2 修为卡统一 → D1 趋势看板 → B1 周快照与周报
第三阶段（深化）：D2 预警闭环 → D3 洞察打通 → B2 闭关清单 → E2 出卷护栏 → C1 仪式感
第四阶段（探索）：A3 材料检验 → A4 互动质量 → B3 传功 → B4 权重配置 → E3 用量看板
```

依赖关系：G1 是 B1/D1/D2/B4 的前置；G2 是 A1（定量版）/C2 日志区/D3 时间线的前置；E1 是 A3/E2/E3 的前置；A2 独立可先行。第四阶段均为可独立交付的探索项，任一项受阻不影响其他。

## 架构原则（贯穿全部条目）

1. **读路径不算、不写、不调 AI**：一切昂贵计算走调度器或本人动作触发，页面只读快照/表。
2. **状态表是真相源**：试炼、预警、任务的状态机落库，执行载体（调度器/后台任务）可替换可恢复。
3. **AI 输出必过确定性校验，且永远有非 AI 降级**：规则文案、行为判定、精确去重是每个 AI 功能的兜底形态。
4. **规则有版本**：权重、判定标准的每次变更记录版本并在流水中可见，历史分数永远可解释。
5. **新表先想归档**：流水、日志、快照类表在建表同时定归档策略（70GB 磁盘约束）。
6. **复用既有基建**：统一调度器、消息中心、learning-path 动作结构、行为事件表、P03 验证 harness——本文档全部条目零新增基础组件。
